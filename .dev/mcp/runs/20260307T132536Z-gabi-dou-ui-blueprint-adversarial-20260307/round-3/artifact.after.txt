# GABI DOU — Blueprint de Frontend v3 (Revisado)

## 1. Conceito Visual: "Brasília Moderna"

Identidade inspirada no racionalismo brasileiro: espaço generoso, tipografia com autoridade, contraste calculado. O dourado do Diário Oficial como acento — não como tema. Nada genérico, nada que pareça template SaaS.

---

## 2. Stack Técnica

| Camada | Escolha | Justificativa |
|---|---|---|
| Framework | React 18 + Vite 5 | Fast refresh, ESM nativo, tree-shaking agressivo |
| Primitives UI | shadcn/ui (Radix) | Copia-se o primitivo, descarta-se o estilo |
| Ícones | Lucide React | Tree-shakeable (~0.5KB/ícone), sem identidade genérica |
| Bottom Sheet | vaul | <5KB gzip, spring physics nativo, acessível |
| Motion | Framer Motion 11 | Apenas para layout animations e page transitions; NÃO para sheets |
| Estilos | Tailwind CSS 4 + CSS custom properties | Tokens como variáveis, utility classes para layout |
| Estado client | Zustand (+ persist middleware com IndexedDB) | Leve, sem boilerplate, async storage via idb-keyval (~600B) |
| Estado server | TanStack Query v5 | Cache, stale-while-revalidate, retry automático, prefetch, AbortSignal |
| Virtualização | react-virtuoso | estimateSize dinâmico para documentos longos, endReached para streaming |
| Busca local | FlexSearch | Busca full-text client-side dentro de documento aberto |
| Strings | `messages/pt-BR.ts` (typed dictionary) | Zero runtime cost, type-safe, trivial migração futura para i18n |
| Observabilidade | @sentry/react + web-vitals | Error tracking com source maps, métricas de performance |
| Testes | Vitest + Testing Library + Playwright + axe-playwright | Unit → component → e2e com a11y automatizado |
| Dev/Doc | Ladle | Storybook alternativo nativo em Vite, desenvolvimento isolado |
| a11y | @axe-core/react (dev) + jest-axe (CI) + NVDA/JAWS manual | Gate automatizado + teste com leitor de tela |
| Bundle Check | size-limit | CI gate para budgets granulares por chunk |

```css
  --text-3xl:  clamp(1.94rem, 1.6rem + 1.7vw, 2.75rem);

  /* Line heights */
  --leading-tight:  1.2;
  --leading-normal: 1.5;
  --leading-relaxed:1.7;

  /* Letter spacing */
  --tracking-tight:  -0.02em;
  --tracking-normal:  0;
  --tracking-wide:    0.04em;
}
```

**Estratégia de loading crítico (LCP < 2.5s):**
1. **Preload crítico:** IBM Plex Sans 400/600, Literata 400/400i (subset latin-ext).
2. **Font-display:** `swap` para display e body (Literata). `optional` apenas para JetBrains Mono.
3. **Mitigação CLS:** `size-adjust: 103%` no @font-face de Literata para reduzir shift de Georgia→Literata.
4. **Fallback:** System serif stack (Georgia) renderiza progressivamente.

**Mapeamento de pesos:**
- Headings: display/600 (semibold)
- Body: body/400 (regular)
- Labels/Captions: display/400 (regular)

## 5. Performance Budget & Estratégia

| Métrica | Target | Enforcement |
|---|---|---|
| Vendor chunk (React+ReactDOM) | ≤ 140 KB gzip | size-limit (custo fixo, monitorar upgrades) |
| Shared libs (TanStack, Zustand, Framer Motion, Radix) | ≤ 50 KB gzip | size-limit |
| App initial route (excl. vendor) | ≤ 50 KB gzip | code-splitting obrigatório |
| Total initial transfer | ≤ 210 KB gzip | Lighthouse CI gate |
| Lazy routes | ≤ 30 KB cada | dynamic imports documentados |
| LCP (3G throttled) | < 2.5s | Lighthouse CI |
| CLS | < 0.05 | font-size-adjust no @font-face |
| INP | < 200ms | |
| Lighthouse Performance | ≥ 90 | |
| Lighthouse Accessibility | ≥ 95 | axe-playwright + manual |

**Code-splitting obrigatório:**
- Rota /dashboard: lazy
- Rota /document/:id: lazy
- DocumentViewer: lazy load de FlexSearch
- JetBrains Mono: lazy (sem preload)

**Prefetch estratégia:**
- Na lista de resultados, prefetch do primeiro documento visível via `queryClient.prefetchQuery`
- Documentos DOU são imutáveis: `staleTime: Infinity` no TanStack Query

## 6. Estado: Boundaries (Arquitetura)

Demarcação rígida de responsabilidades para evitar sincronização incorreta:

| Tipo | Tecnologia | Responsabilidade | Persistência |
|---|---|---|---|
| **Server State** | TanStack Query | Dados DOU, resultados de busca, seções de documentos | Cache em memória (staleTime) |
| **Client Preferences** | Zustand + IndexedDB | Tema, fontSize, preferências de leitura | IndexedDB (async, não bloqueante) |
| **Session State** | Zustand (sem persist) | Filtros ativos, aba selecionada, estado de UI transitório | Memória apenas |
| **Navigation State** | URL (hash + query) | scrollPosition, activeSection, searchQuery, page | URL (shareable, back/forward) |

**Regras:**
- Nunca persistir scrollPosition no Zustand (usar URL hash)
- Nunca armazenar dados DOU no Zustand (usar TanStack Query)
- Migrações de schema Zustand: usar opção `version` + `migrate` desde dia 1

## 7. Acessibilidade (WCAG 2.1 AA + LBI 13.146/2015)

- **Contraste:** Todos os pares foreground/background validados (Seção 3.1). Accent text usa tokens específicos por tema.
- **Navegação:** Skip-to-content link, landmarks ARIA, roving tabindex em listas virtualizadas.
- **Motion:** `prefers-reduced-motion` respeitado em todos os tokens.
- **Zoom:** Suporte a 200% zoom sem quebra (breakpoints fluidos).
- **Leitores de tela:** 
  - Anúncio de resultados dinâmicos via `aria-live="polite"`
  - Highlights no documento com `role="mark"` e `aria-label` indicando ocorrência
  - Teste manual obrigatório com NVDA/JAWS antes de release
  - Testes automatizados: axe-playwright no CI
- **Alto contraste:** Tema `high-contrast` completo com todos os tokens semânticos.
- **Foco:** Focus ring visível (`--color-focus-ring`), trap de foco em modais/sheets, restauração de foco ao fechar.

## 9. Architecture Decision Records (ADRs)

### ADR-001: BottomSheet Implementation
**Decisão:** Usar `vaul` (emilkowalski/vaul) em vez de Framer Motion custom.
**Rationale:** 
- vaul é específico para sheets, <5KB gzip, implementa focus trap, snap points e gestures nativamente
- Usa primitives próprios baseados em patterns WAI-ARIA, sem dependência de Radix Dialog
- Compatível com a stack pois shadcn/ui também usa Radix primitives independentes
- Framer Motion reservado apenas para layout animations e page transitions

### ADR-002: Estratégia de Temas
**Decisão:** Definir dark, light e high-contrast em Fase 0, mesmo que UI não exponha toggle inicial.
**Rationale:** 
- Evita dívida técnica de migração posterior
- Garante que nenhum componente use cores hardcoded
- Conformidade com LBI 13.146/2015 desde o início
- High-contrast inclui todos os tokens semânticos (danger, success, info, etc.)

### ADR-003: Loading de Documentos
**Decisão:** DocumentViewer usa streaming via `contentLoader` callback + react-virtuoso, nunca carrega documento completo via prop.
**Rationale:**
- Documentos DOU podem ter centenas de artigos
- Virtualização de renderização é inútil se fetch é monolítico
- Suporta lazy loading de seções e AbortSignal para cancelamento
- react-virtuoso escolhido sobre @tanstack/react-virtual por melhor estimativa de altura dinâmica para conteúdo misto (texto + tabelas + imagens)

### ADR-004: Internacionalização (i18n)
**Decisão:** Usar typed dictionary (`messages/pt-BR.ts`) em vez de react-intl desde o início.
**Rationale:**
- Aplicação é monolíngue pt-BR por natureza regulatória (DOU)
- Zero runtime cost (react-intl adiciona ~10-30KB)
- Type-safe keys, single source of truth
- Trivial migração para react-intl se requisito de i18n surgir no futuro (Fase 4)
- Evita premature abstraction — a segunda forma mais cara de dívida técnica

### ADR-005: Estado e Persistência
**Decisão:** Separação estrita entre TanStack Query (server), Zustand (client), e URL (navigation).
**Rationale:**
- Evita sincronização dupla entre caches
- scrollPosition/activeSection em URL permite deep-link e back/forward functional
- Zustand persist usa IndexedDB (async) em vez de localStorage (sync) para não bloquear main thread
- Schema versioning desde dia 1 para migrações de preferências do usuário
