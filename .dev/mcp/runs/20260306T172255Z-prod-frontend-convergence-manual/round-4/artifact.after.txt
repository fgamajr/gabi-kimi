0. PRE-MIGRATION AUDIT & BACKEND ENDPOINT MAPPING
═══════════════════════════════════════════════════════════════

Before any frontend code is written, execute this checklist:

Step 1: Audit web_server.py routes.
  - List every route handler in web_server.py.
  - Map each existing endpoint to the api.js fetch function it will serve.
  - If backend uses /api/v1/search, api.js calls /api/v1/search — the frontend adapts to the backend, never the reverse.
  - Document the exact mapping in a ROUTES.md file at repo root.

Step 2: Confirm static-file serving behavior.
  - Verify web_server.py serves index.html for GET / with no path-based routing for sub-pages.
  - Confirm it serves static assets from src/frontend/web/ (CSS, JS, fonts, images).
  - If web_server.py has catch-all or path-based HTML serving, document it — this affects routing strategy.

Step 3: Verify API health endpoint exists.
  - If GET /api/health (or equivalent) exists, map it. If not, flag for backend team to add.
  - api.js uses this on Shell mount to display connectivity warnings (non-blocking — see Section 5).

Step 4: Runtime 404 detection — distinguish dev errors from data 404s.
  - api.js must log a clear console.error with the endpoint path when it receives a 404.
  - API-route 404 message: "Endpoint not found: GET /api/foo — verify backend route mapping in ROUTES.md"
  - Resource 404 (e.g., /api/docs/:id where :id doesn't exist): render <ErrorState type="not-found" message="This item may have been deleted" /> — no retry button.
  - Dev-error 404 (endpoint itself missing): render <ErrorState type="endpoint-missing" message="Feature unavailable" onRetry /> — retryable.
  - Distinguish by convention: if the route pattern matches a known endpoint template but the specific resource ID fails, it's a resource 404. If the route pattern itself is unknown, it's a dev error.

Step 5: Verify Content-Type headers.
  - Confirm web_server.py serves .js files with Content-Type: application/javascript (not text/plain).
  - Confirm .css files with Content-Type: text/css.
  - Confirm .woff2 files with Content-Type: font/woff2.
  - Some Python HTTP servers (http.server, SimpleHTTPServer) serve incorrect MIME types.
    Browsers refuse ES module execution with strict MIME type errors — this is a silent, hard-to-diagnose failure.
  - If MIME types are wrong, fix in web_server.py before proceeding.

Migration sequence (big-bang replacement — no feature flags):
  Step M1: Copy current index.html → _legacy/index-alpine.html (preservation, not coexistence).
  Step M2: Create new index.html with React mount point, theme script, CSS links.
  Step M2.5: Create package.json in src/frontend/web/ with:
    dependencies: react@18, react-dom@18
    devDependencies: esbuild@0.20+
    scripts:
      "build": "esbuild src/main.jsx --bundle --format=esm --outdir=dist --minify --target=safari15,chrome89,firefox90 --metafile=meta.json --jsx=automatic"
      "dev": "esbuild src/main.jsx --bundle --format=esm --outdir=dist --target=safari15,chrome89,firefox90 --jsx=automatic --sourcemap=inline --watch"
      "check-budget": "node scripts/check-bundle-size.js"
    Run npm install. Commit package.json and package-lock.json.
    Add node_modules/ to .gitignore.
  Step M3: Run npm run build → dist/main.js. web_server.py serves new index.html at / unchanged.
  Step M3.5: Verify MIME types (Step 5 above) by loading dist/main.js in browser — check DevTools Network tab for Content-Type and console for MIME errors.
  Step M4: Smoke-test all API routes from new frontend against running backend.
  Step M5: Remove _legacy/ after 2 weeks of production stability.
  Rollback: revert index.html to _legacy/index-alpine.html. Zero backend changes required.

1. BUILD SYSTEM & MODULE STRATEGY
═══════════════════════════════════════════════════════════════

Build tool: esbuild (single dependency, zero-config).

Decision: esbuild bundles React into the output. No vendor/ directory. No import maps.
Rationale: import maps require Safari 16.4+ / Firefox 108+ minimum. Bundling React eliminates
the compatibility risk entirely and simplifies the build to a single command. The bundle size
difference is zero since React ends up in the output either way.

Production command:
  esbuild src/frontend/web/src/main.jsx \
    --bundle \
    --format=esm \
    --outdir=src/frontend/web/dist \
    --minify \
    --target=safari15,chrome89,firefox90 \
    --metafile=meta.json \
    --jsx=automatic

Development command:
  esbuild src/frontend/web/src/main.jsx \
    --bundle \
    --format=esm \
    --outdir=src/frontend/web/dist \
    --target=safari15,chrome89,firefox90 \
    --jsx=automatic \
    --sourcemap=inline \
    --watch

  Dev workflow: run `npm run dev` in one terminal (esbuild watch rebuilds on file change).
  Run web_server.py in another terminal. Both serve from the same directory.
  No proxy needed — esbuild writes to dist/, web_server.py serves the whole web/ directory.
  API requests go to web_server.py directly (same origin, no CORS).

--jsx=automatic rationale: enables React 18's automatic JSX runtime. Components do NOT need
`import React from 'react'` at the top. esbuild injects jsx-runtime imports automatically.
Without this flag, any component missing the React import silently fails at runtime.

Target rationale: using explicit browser targets instead of es2020 lets esbuild make
optimal per-browser syntax decisions. This is self-documenting of the actual support matrix.

IMPORTANT: esbuild --target downlevels SYNTAX only, not runtime APIs.
Banned APIs (not available on browser floor):
  - AbortController.timeout() — Safari 16+ only
  - Array.prototype.at() — Safari 15.4+ only (borderline, avoid)
  - Object.hasOwn() — Safari 15.4+ only
  - structuredClone() — Safari 15.4+ only
  - crypto.randomUUID() — Safari 15.4+ only
  Add this as a lint rule or PR checklist item. Use Object.prototype.hasOwnProperty.call() etc.

CSS strategy: tokens.css and animations.css are NOT imported by JS. They are standalone
<link> references in index.html, loaded directly by the browser. esbuild does not process them.
This is intentional — global design tokens and keyframes are authored CSS, not component CSS.

No --splitting in v1 (splitting requires import maps for shared chunks in browsers;
we just eliminated import maps). Single entry bundle. Code-split in v2 via dynamic import()
if bundle exceeds budget.

Bundle budget enforcement:
  CI step parses meta.json, fails build if dist/main.js gzipped > 80KB.
  Budget breakdown (estimated):
    React 18 + ReactDOM:  ~42KB gzipped
    App components:       ~15KB gzipped
    Icons (tree-shaken):  ~3KB gzipped
    CSS (tokens + anim):  ~4KB gzipped (loaded separately, not in JS budget)
    Total JS:             ~60KB gzipped (20KB headroom to 80KB cap)

Preact escape hatch: if bundle exceeds 80KB, switch via esbuild alias:
  --alias:react=preact/compat --alias:react-dom=preact/compat
  Preact compat is ~4KB vs React's 42KB. Document this as v1.1 option, not default.

Linting (non-negotiable for v1):
  devDependencies: eslint, eslint-plugin-react-hooks
  .eslintrc: { "plugins": ["react-hooks"], "rules": { "react-hooks/rules-of-hooks": "error", "react-hooks/exhaustive-deps": "warn" } }
  CI runs eslint before build. This catches stale closures and missing dependencies.

Core Web Vitals targets (non-negotiable):
  LCP: < 2.5s on 3G throttled
  FID / INP: < 100ms
  CLS: < 0.1
  Implementation requirements to hit these:
    - index.html inlines critical CSS for above-fold content (shell chrome, skeleton placeholders).
    - tokens.css and animations.css loaded via <link rel="stylesheet"> (not @import — avoids waterfall).
    - Fonts loaded via <link rel="preload" as="font" type="font/woff2" crossorigin> with font-display: swap.
    - All above-fold layout uses explicit dimensions (no layout shift from late-loading content).
    - Skeleton placeholders match final content dimensions to prevent CLS on data arrival.

Source code lives in src/frontend/web/src/ (JSX, full React idioms).
Built output lands in src/frontend/web/dist/ (static ES module, minified, tree-shaken).
index.html references dist/main.js via <script type="module">.
web_server.py serves src/frontend/web/ unchanged — only static file additions.

Browser support floor: Chrome 89+, Safari 15+, Firefox 90+, Edge 89+.

3. NAVIGATION & INFORMATION ARCHITECTURE
═══════════════════════════════════════════════════════════════

Bottom tab bar: 3 tabs (Home, Search, Library).
  - Tabs are parallel peer destinations. No drill-down views in tabs.
  - Semantic markup: <nav aria-label="Main"> with <a> elements, NOT role="tablist".
    Rationale: these are route-level navigations that change the URL, not panel switches.
    Active link gets aria-current="page". Inactive links have no aria-current.
  - Active tab visual: accent-colored filled icon + label text + 4px bottom border (--accent-default).
    Inactive: outline-stroke icon + --text-tertiary label. Both color AND weight differentiation required.
  - Tab bar icon size: 24px. Label font: --font-size-xs (12px). Tab item height: 56px + safe area.
  - Pointer device hover state: --surface-2 background on tab item (subtle, not accent).
  - Tab bar receives `inert` attribute + `aria-hidden="true"` when bottom sheet is open.
    CSS: `[inert] { visibility: hidden; }` (prevents screen reader and keyboard access).
    Main content area ALSO receives `inert` + `aria-hidden="true"` when bottom sheet is open.
    Focus returns to the trigger element that opened the sheet on close.
  - Tab bar is HIDDEN on Detail routes: display:none + inert.
    Shell derives: `const showTabBar = ['/', '/search', '/library'].includes(currentPath)`
    Detail screen has its own StickyHeader with back navigation — no tab bar needed.

  Tab switch history behavior: REPLACE, not PUSH.
    Tab switches use history.replaceState() — they do NOT create history entries.
    Only drill-down navigations (Home→Detail, Search→Detail) use history.pushState().
    This prevents the "history trap": Home → Detail → Search (replace) → Back returns to Detail → Back returns to Home.
    Correct behavior: Back from a tab always returns to the previous drill-down context, not to another tab.

  iOS safe area: Tab bar applies padding-bottom: env(safe-area-inset-bottom, 0px).
    index.html includes <meta name="viewport" content="width=device-width, initial-scale=1, viewport-fit=cover">.
    --safe-bottom: env(safe-area-inset-bottom, 0px) defined in tokens.css.

Drill-down views (Detail/Timeline): accessed from tapping items on Home or Search results.
  - When route matches #/doc/:id: all three persistent tab panels receive hidden+inert.
    DetailScreen mounts in the same <main> slot. Tab bar is hidden (see above).
  - Back affordance: chevron-left button in top-left header area (44×44px tap target).
  - Push onto history via pushState (not replaceState).
  - Deep link handling: when DetailScreen mounts and detects direct entry (no history.state
    or the router's internal stack is empty), the back button renders as a Home link (href="#/")
    instead of calling history.back(). This prevents stranding users at a dead-end.

Routing strategy: hash-based (#/) for zero-config static serving.
  Confirmed: web_server.py is a static file server. It does NOT serve route-specific HTML.
  Hash routing is an intentional v1 decision.
  v2 escape hatch: migrate to History API pushState + web_server.py catch-all that serves
  index.html for all non-API, non-static paths. Document in ARCHITECTURE.md.

  CRITICAL IMPLEMENTATION NOTE — Hash query param parsing:
    Hash query params are NOT accessible via location.search. The browser sees everything
    after # as the fragment. The Router MUST parse location.hash manually:

      function parseHash(hash) {
        const raw = hash.slice(1) || '/';
        const [path, queryString] = raw.split('?');
        const query = queryString
          ? Object.fromEntries(new URLSearchParams(queryString))
          : {};
        return { path: path || '/', query };
      }

    If someone implements useRoute() by reading new URL(location).searchParams, every
    preset-initiated search will silently fail with an empty query.

    Edge cases to test:
      '#'           → { path: '/', query: {} }
      '#/'          → { path: '/', query: {} }
      '#/search'    → { path: '/search', query: {} }
      '#/search?q=test'        → { path: '/search', query: { q: 'test' } }
      '#/search?q=test&page=1' → { path: '/search', query: { q: 'test', page: '1' } }
      '#/search?q=C%23+prog'   → { path: '/search', query: { q: 'C# prog' } }

    Known limitation — hash fragments containing literal # characters:
      Hash fragments with literal # are ambiguous per RFC 3986. The browser truncates
      location.hash at the second #. SearchComposer MUST encodeURIComponent the query
      value before writing to location.hash (encodes # as %23). Decoding happens
      automatically via URLSearchParams. Add a test case for queries containing #, &, = chars.

  Routes:
    #/              → Home screen
    #/search        → Search screen (reads ?q= param for preset-initiated searches)
    #/library       → Library screen
    #/doc/:id       → Detail/Timeline screen (drill-down, not a tab)

  Base path variable: const BASE = '' (empty string). If app moves to /app/ subpath,
  change to '/app' and all route matching prepends BASE. Defined once in navigation.jsx.

  state/navigation.jsx exports:
    RouterContext: { path, params, query, push(path), replace(path), back(), isDirectEntry }
    useRoute(): reads context
    <Router>: listens to hashchange, parses hash path + query params (using parseHash above),
      updates context, announces to route-announcer, manages scroll restoration

  Focus management on route change:
    - Each screen's root container has tabindex="-1" (enables programmatic focus).
    - On route change, focus moves to the screen container element.
    - :focus-visible styles apply; :focus:not(:focus-visible) gets outline: none (no ugly ring on mouse click).
    - RouteAnnouncer speaks "Navigated to {page name}" via aria-live="assertive".

  RouteAnnouncer debounce:
    - 150ms debounce on announcement updates. If navigation occurs again within 150ms,
      cancel the previous announcement.
    - Single aria-live region with aria-atomic="true".
    - Implementation: useEffect with setTimeout(150ms), clear on cleanup or re-fire.

  Scroll restoration:
    - On pushState (drill-down): store window.scrollY in a bounded LRU Map keyed by outgoing route path.
    - LRU Map: max 20 entries. On insert at capacity, delete the oldest (first) key.
    - On back navigation: restore scrollY from the Map after target screen renders (requestAnimationFrame).
    - On replaceState (tab switch): scroll to 0 (tabs manage their own scroll internally).
    - The Map lives in memory. Scroll positions are transient. Deliberately NOT sessionStorage —
      stale scroll positions after data changes cause worse UX than resetting to top.

Search presets on Home: tapping a preset navigates to #/search?q={encodeURIComponent(preset.query)}.
  SearchScreen reads ?q= from the hash query on mount and auto-executes the search.
  The Search tab becomes visually active. The preset is a shortcut, not a duplicate search surface.

  Manual search URL sync: SearchScreen uses history.replaceState (never pushState) to update
  #/search?q={encodeURIComponent(value)} as the user types, debounced at 300ms. This ensures:
    - Search is shareable/bookmarkable at any point.
    - Refresh preserves the current query.
    - History is NOT polluted with per-keystroke entries.

Lazy-Mount Tab Panels (state preservation without DOM persistence):
  DECISION: Lazy mount + state caching, NOT simultaneous DOM mount.
  Rationale: mounting all 3 tabs simultaneously fires 3x API calls on app init, retains 3x
  component trees in memory, and wastes resources on screens the user may never visit.
  Low-end mobile devices (<4GB RAM) will suffer.

  Implementation:
    - Each tab screen mounts on FIRST VISIT, not on app init.
    - Once mounted, the tab stays mounted for the session (display:none + inert when inactive).
    - A tab that has NEVER been visited does not exist in the DOM.
    - Shell tracks: `const [mountedTabs, setMountedTabs] = useState(new Set(['home']))`.
      Home is pre-mounted (it's the default route). Search and Library mount on first navigation.
    - On tab switch: add tab to mountedTabs Set if not present, then toggle visibility.
    - Each tab's scroll container retains its scrollTop because the DOM node persists.
    - API calls fire only when the tab mounts for the first time.

    v1.1 enhancement (document, do not implement): if profiling shows memory pressure,
    add a keepalive timeout — unmount a tab if user hasn't visited it in 10 minutes,
    with state snapshot to useRef before unmount.

  When route is #/doc/:id (Detail):
    - All mounted tab panels receive hidden+inert.
    - DetailScreen mounts in the same <main> slot.
    - Tab bar hidden (showTabBar = false).
    - On back navigation, the previous tab's panel becomes visible again (already mounted).

Bottom sheet:
  - Invoked from contextual buttons (not a tab destination).
  - Container attributes: role="dialog", aria-modal="true", aria-labelledby={titleId}.
  - Opens over content with overlay (--surface-0 at 60% opacity).
  - Drag handle: 44×44px interactive area containing a 32px wide × 4px tall visual indicator,
    --surface-3 background, --radius-full, centered with var(--space-4) top margin.
    The 44×44px area is the touch target; the 4×32px bar is the visual affordance.
  - Dismiss: drag down past 20% of sheet height OR downward velocity > 0.5px/ms, Escape key, overlay tap.
  - Focus trapped inside while open (FocusTrap component).
  - When sheet opens: TabBar AND <main> receive `inert` + `aria-hidden="true"`.
    Legacy fallback: also add aria-hidden="true" to all direct children of <body> except
    the portal root. This ensures VoiceOver on iOS 14/15 (which doesn't support inert) correctly
    limits its navigation scope.
  - On close: remove all inert/aria-hidden attributes. Check triggerRef.current?.isConnected
    before returning focus. If trigger is disconnected, fall back to
    document.querySelector('[role="main"]')?.focus() or the first focusable element in viewport.

Quick actions on list items: explicit "⋯" (more) icon button (44×44px tap target). No long-press. No swipe gestures in v1.

Pull-to-refresh: deferred to v2. v1 uses manual refresh via error-state retry buttons and
status card's automatic polling. overscroll-behavior-y: contain applied to main scroll container
to prevent browser default pull-to-refresh (which triggers full page reload and is worse than no PTR).
