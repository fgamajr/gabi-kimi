# Frontend Audit: DOU Research Tool ŌĆö Iteration Review

## Executive Summary

This iteration delivers real structural progress ŌĆö shared app shell, query transparency, and a first normative-awareness layer ŌĆö but the product remains a **document browser with investigation aspirations**, not yet an investigation tool. Four areas carry design decisions that are actively dangerous for the target user base: unsanitized document rendering, heuristic-based legal validity without adequate mitigation, auto-applied inferred search filters, and unaddressed accessibility. These must be corrected before any production exposure to legal/audit professionals.

---

## 1. What Actually Improved (Capability Gains, Not Cosmetic)

### Shared App Shell + Route-Level Layout
**Real gain.** Persistent navigation rail (desktop) and bottom nav (mobile) establish consistent wayfinding across search and reading workflows. Route-level layout control is architecturally sound ŌĆö it allows document routes to manage their own chrome without forking the shell. This is infrastructure that compounds. However, as noted in Blocker 3, this architectural choice creates a tension with investigation-loop navigation that must be resolved explicitly rather than papered over with shell-imposed chrome.

### Query Transparency via interpreted_query / inferred_filters / applied_filters
**Real gain.** Surfacing the system's interpretation of a query is a prerequisite for professional trust. Legal researchers must know *exactly* what the system searched for. The separation into three distinct layers (raw interpretation, system inference, user-confirmed filters) is the correct data model. However, the **presentation** of this model has a critical flaw (see Section 2, Blocker 2).

### Normative Timeline Derived from References
**Partial gain.** Showing a timeline of connected normative references is useful orientation. But the value is capped by the heuristic validity layer it depends on ŌĆö a timeline built on uncertain data is a timeline that cannot be cited.

## 2. What Still Blocks Production Usability

### BLOCKER 1 ŌĆö Heuristic Normative Status Is a Liability, Not a Feature

**Severity: Critical ŌĆö production gate until Phase A mitigation is implemented.**

The "Situa├¦├Żo normativa" card uses "cautious heuristics" to infer revocation, replacement, alteration, and complementation signals. For legal and audit professionals ŌĆö the stated target ŌĆö this is worse than showing nothing.

**Why it's dangerous:** A legal professional who sees a status card that says "revogado" (revoked) or shows no revocation signal will treat that as authoritative unless the UI aggressively communicates otherwise. A false negative (showing a revoked law as active) can lead to reliance on invalid legislation. A false positive (showing a valid law as revoked) can cause a researcher to miss controlling authority.

**Severity/timeline reconciliation:** "Critical" means this blocks production. Phase A below is not "ship with warnings" ŌĆö it is "remove the dangerous surface entirely and replace it with a safe alternative." Phase A is the production gate. Phase B is the feature restoration.

**Required correction ŌĆö phased approach acknowledging backend dependency:**

**Phase A ŌĆö Immediate (1ŌĆō2 weeks, frontend only, production gate):** Do not display the ValidityCard for heuristic data. The prominent status card UI is reserved for when verified data is actually available. Instead, render only a subtle document footer element: a single line reading "Situa├¦├Żo normativa: consultar fonte oficial" linked to the authoritative Planalto URL. No heuristic signals, no status determination, no color-coded banners. This avoids training users to ignore warnings (alarm fatigue) and avoids presenting unverified legal determinations in any form.

**Phase B ŌĆö Epic (3ŌĆō6 months, backend pipeline):** Integration with Planalto's consolidation database (SILEG) and LexML, with conflict resolution, retroactive modification handling, and temporal graph maintenance. Only after this pipeline is operational does the ValidityCard become visible.

**Residual risk disclosure (must appear in product documentation and terms of use):** Even after Phase A, the tool does not provide authoritative normative status information. Users must independently verify the current status of any legislation through official sources. This residual risk exists until Phase B is complete and must be documented in the product's terms of service, not just the UI.

**ValidityCard component specification (built during Phase A, rendered only when Phase B data is available):**

```typescript
type ConfidenceLevel = 'verified' | 'unverified';

interface ProvenanceInfo {
  sourceUrl: string;          // e.g., Planalto URL, LexML URN
  sourceLabel: string;        // e.g., "Planalto ŌĆö Legisla├¦├Żo Consolidada"
  syncTimestamp: string;      // ISO 8601
  contentHash: string;        // SHA-256 of source document at sync time
  verificationMethod: string; // e.g., "sileg_api" | "lexml_crossref"
}

interface ValidityCardProps {
  confidenceLevel: ConfidenceLevel;
  status: string | null;              // "vigente" | "revogado" | "alterado" | null
  signals: NormativeSignal[];
  provenance: ProvenanceInfo;         // required ŌĆö no card without provenance
  officialSourceUrl: string;          // always present ŌĆö link to authoritative source
}
```

**Rendering rules:**

| Level | Visibility | Banner | Status display | Provenance footer |
|---|---|---|---|---|
| `verified` | Visible | Green ŌĆö "Verificado em [source] em [date]" | Displayed as determined status | Full provenance: source, sync timestamp, hash, method |
| `unverified` | **Hidden** ŌĆö component does not render. Footer link to official source only. | N/A | N/A | N/A |

**Phase A ships with the ValidityCard component built but gated on `metadata_quality: 'authoritative'` from the API response.** No document in Phase A will trigger card rendering. The footer link to the official source is always present regardless of metadata quality.

**Backend API contract required:**

```typescript
interface NormativeMetadataResponse {
  metadata_quality: 'authoritative' | 'heuristic' | 'none';
  status: string | null;
  signals: NormativeSignal[];
  provenance: ProvenanceInfo | null;
  official_source_url: string;
}
```

### BLOCKER 2 ŌĆö Inferred Filters Auto-Applied Without User Confirmation

**Severity: High.**

The search interface currently surfaces `inferred_filters` as applied constraints alongside user-selected `applied_filters`. In legal discovery and audit workflows, the distinction between "what I asked for" and "what the system decided I meant" is the difference between a defensible search methodology and a negligent one.

**Why it's dangerous:** If a system silently narrows a search ŌĆö for example, interpreting "decreto 2018" as `type:decreto AND year:2018` when the user intended a keyword search across all document types mentioning "decreto 2018" ŌĆö it can exclude relevant results. The user cannot verify completeness of their research if the system is silently shaping their query.

**Required correction:**
- `inferred_filters` must be surfaced as **suggested filter chips** in a visually distinct "Sistema sugeriu" group, not auto-applied to the query.
- User must explicitly accept each inferred filter (tap/click to promote to `applied_filters`) or dismiss it.
- Once accepted, the filter moves to the standard applied filter chip group with no visual distinction ŌĆö it's now the user's filter.
- The `interpreted_query` display should remain visible as-is (showing raw system interpretation is good transparency), but it must be clearly labeled as diagnostic, not authoritative.

**Query Audit Log ŌĆö local-only, no encryption:**

Every search session should be exportable as a structured record for legal research reproducibility. However, the previous iteration's client-side encryption proposal was security theater: storing an encryption key alongside encrypted data on the same client device provides no additional protection over plaintext client-side storage, while introducing key management failure modes (key loss = permanent audit log loss).

```typescript
interface QueryAuditRecord {
  sessionId: string;                  // UUID, generated client-side
  timestamp: string;                  // ISO 8601
  methodology: {
    inferredFilters: FilterSpec[];
    userDecisions: { filter: FilterSpec; action: 'accepted' | 'rejected' }[];
    appliedFilters: FilterSpec[];
    resultCount: number;
  };
  content: {                          // client-side only ŌĆö NEVER transmitted to server
    rawQuery: string;
    interpretedQuery: string;
  };
}

interface QueryAuditExport {
  format: 'json' | 'csv' | 'pdf';
  includeContent: boolean;            // user chooses: full export or methodology-only
  records: QueryAuditRecord[];
  exportTimestamp: string;
  toolVersion: string;
}
```

**Storage and privacy requirements:**
- All query audit data is stored **client-side only** in IndexedDB. The `content` block is never transmitted to any server endpoint. No server-side persistence of raw query text.
- Auto-expiry: 30 days by default, user-configurable per session. Implemented via IndexedDB record TTL checked on app initialization.
- Export: "Export to JSON/CSV/PDF" function allows user backup. On first export, prompt user to choose between full export (includes search terms) or methodology-only export (omits `content` block). This choice is the privacy control ŌĆö no encryption complexity.
- If server-side persistence is added in a future iteration, it must use server-side encryption with user authentication, data subject access/deletion endpoints (LGPD), and explicit opt-in consent. Do not retrofit client-side encryption.

### BLOCKER 3 ŌĆö Mobile Navigation on Document Routes Breaks Investigation Loop

**Severity: Medium, but high-friction for the core workflow.**

Legal research is not linear reading ŌĆö it's a rapid oscillation between search results and document inspection. Hiding mobile bottom nav on document routes forces reliance on browser back-button, which destroys filter state and scroll position in the results list.

**Architectural clarification:** The route-level layout autonomy praised in Section 1 is the right architecture. The solution must not contradict it by forcing shell-imposed chrome onto document routes. Instead, the navigation problem should be solved at the state management layer.

**Required correction ŌĆö two complementary measures:**

1. **Search state preservation (mandatory, solves the root cause):** Search state (query, filters, scroll position, result set page) must be preserved in URL search parameters with sessionStorage fallback. Use `History.replaceState()` for filter changes within the same search session (prevents history pollution from every filter toggle). Use `History.pushState()` only for new search submissions (creates a meaningful back-navigation point). **URL length limits:** Complex filter combinations can exceed browser URL length limits (~2KB safe maximum). When serialized state exceeds 1.5KB, store full state in sessionStorage keyed by a short UUID, and encode only the UUID reference in the URL parameter (`?sid=abc123`). This preserves deep-linkability for simple queries while preventing silent state loss on complex ones.

2. **Document route opt-in navigation (respects route autonomy):** Document routes *choose* to render a lightweight context bar (not imposed by the shell). This bar is part of the document route's own layout, showing: [ŌåÉ Resultados (N)] | [Filter badge count]. The document route requests this from a shared component; the shell does not inject it. This preserves the architectural principle while providing wayfinding.

3. **Tablet breakpoint (768pxŌĆō1024px):** Define explicit behavior ŌĆö tablet uses the desktop navigation rail in collapsed (icon-only) mode. Do not leave this breakpoint undefined; it's a common device class for legal professionals using iPads in courtrooms and audit rooms.

### BLOCKER 4 ŌĆö Unsanitized Document Rendering Creates XSS Vectors

**Severity: Critical.**

The document reader renders HTML content scraped from DOU. DOU documents contain complex HTML tables, inline styles, embedded metadata, and potentially malformed markup. A compromised scrape pipeline, a malicious document injection, or even a legitimate DOU publication with unusual markup could execute scripts in the context of a legal research session, exposing privileged client data, session tokens, or enabling session hijacking.

**Why it's critical for this specific product:** The target users ŌĆö attorneys and auditors ŌĆö handle privileged information. An XSS vector in a document reader they trust becomes an attack surface against attorney-client privilege. This is not a theoretical concern; government gazette scrapers are high-value targets for supply chain attacks.

**Required correction:**

1. **Server-side sanitization at ingest time (defense layer 1):** All scraped HTML must be sanitized through a whitelist-based sanitizer before storage. Allowed elements: `table`, `thead`, `tbody`, `tr`, `th`, `td`, `p`, `br`, `strong`, `em`, `b`, `i`, `u`, `sup`, `sub`, `ol`, `ul`, `li`, `h1`ŌĆō`h6`, `span`, `div`, `caption`, `colgroup`, `col`. Allowed attributes: `colspan`, `rowspan`, `class`, `id`. **Inline style handling:** Strip all raw `style` attributes. During sanitization, extract inline styles from DOU table markup (primarily `width`, `border`, `padding`, `text-align` on table elements), map them to a set of predefined CSS utility classes (e.g., `dou-col-w25`, `dou-text-center`, `dou-border-1`), and replace the inline styles with these class references. Store both raw (for re-processing) and sanitized (for serving) versions. Strip all `on*` event handlers, `href="javascript:..."`, `<script>`, `<iframe>`, `<object>`, `<embed>`, `<form>`, and `data-*` attributes.

2. **Client-side DOMPurify (defense layer 2):** Before rendering any document HTML in the reader component, pass through DOMPurify with a restrictive config matching the server-side whitelist. This is defense-in-depth against server-side sanitizer bypass or pipeline compromise.

3. **Content Security Policy headers ŌĆö no `unsafe-inline`:**
   ```
   Content-Security-Policy:
     default-src 'self';
     script-src 'self';
     style-src 'self';
     img-src 'self' data: https://www.in.gov.br;
     frame-ancestors 'none';
     base-uri 'self';
   ```
   Because inline styles are converted to CSS classes during server-side sanitization (step 1), `'unsafe-inline'` is not needed for `style-src`. This eliminates the CSS injection attack vector (attribute selector exfiltration via `background-image` URLs, CSRF token theft via CSS selectors). `script-src` must NOT include `'unsafe-inline'` or `'unsafe-eval'`.

   **Fallback if class-mapping is incomplete:** If edge-case DOU documents require inline styles that cannot be mapped to utility classes, compute `sha256` hashes of each allowed inline style string during server-side sanitization and inject those hashes into the CSP header dynamically per document response. This is more complex than class mapping but strictly safer than `'unsafe-inline'`. Implement class mapping first; hash-based fallback only for verified edge cases.

4. **Subresource Integrity (SRI)** for all static JS/CSS assets and any external dependencies (fonts, analytics libraries) served from CDN. Legal professionals' browsers must not execute code from compromised CDNs.

5. **Automated XSS regression testing:** Maintain a test suite of known-malicious HTML patterns (reference OWASP XSS Prevention Cheat Sheet and XSS Filter Evasion Cheat Sheet) that must fail sanitization. Run this suite as a CI gate on every change to the ingest pipeline or sanitization configuration. Include at minimum: script injection, event handler injection, CSS injection via `style` attributes, SVG-based XSS, and malformed HTML that could cause parser differential attacks between server sanitizer and browser DOM parser.

## 3. Depth Assessment: Are Search Filters and Normative Awareness Investigation-Grade?

### Search Filters: Structured but Shallow

The current filter set (section, act type, issuing organ, date range) covers the basic facets of DOU document metadata. This is necessary but insufficient for professional legal search.

**What's missing for investigation-grade search:**

1. **Citation parsing in the search bar.** Legal professionals navigate by citation, not by keyword. This is the single highest-impact search UX improvement for the target audience. Current dropdown-based decomposition of citations creates unacceptable friction.

   **Citation parser specification:**

   Brazilian legal citations are highly variable. The parser must handle these formats:

   | Format | Example | Confidence |
   |---|---|---|
   | Compact with year | `Lei 13.709/2018` | High ŌĆö parse to type + number + year |
   | With "n┬║" or "n." | `Lei n┬║ 13.709, de 14 de agosto de 2018` / `Lei n. 13.709/2018` | High ŌĆö parse to type + number + date/year |
   | Hyphenated type | `Decreto-Lei 2.848/1940` | High ŌĆö parse to compound type + number + year |
   | Abbreviated type | `IN RFB 1.234/2020` / `MP 885/2019` | Medium ŌĆö abbreviation is ambiguous (multiple issuing organs for IN; MP = Medida Provis├│ria) |
   | Popular name | `LGPD` | Low ŌĆö requires lookup table, not parsing |
   | Partial | `Decreto 2018` | Ambiguous ŌĆö could be decree #2018 or year filter |

   **Grammar (expanded PEG notation):**
   ```
   Citation     ŌåÉ TypeToken Sep? NumberPrefix? Sep? NumberToken (Sep? "/" Sep? YearToken)?
   TypeToken    ŌåÉ CompoundType / SimpleType / AbbrevType
   CompoundType ŌåÉ ("Decreto" / "Instru├¦├Żo") "-" ("Lei" / "Normativa" / ...)
   SimpleType   ŌåÉ "Lei" / "Decreto" / "Portaria" / "Resolu├¦├Żo" / "Medida Provis├│ria" / ...
   AbbrevType   ŌåÉ "IN" / "MP" / "RDC" / "RE" / "ADI" / ...
   NumberPrefix ŌåÉ "n┬║" / "n." / "n┬░" / "no" / "N┬║" / "N."
   Sep          ŌåÉ [ \t]+
   NumberToken  ŌåÉ [0-9] ("." [0-9]{3})* / [0-9]+
   YearToken    ŌåÉ [0-9]{4}
   ```

   **Fuzzy matching fallback:** If `TypeToken` does not match any known type exactly, compute Levenshtein distance against the canonical type list. Accept if distance Ōēż 2 and the closest match is unambiguous (single candidate). This handles typos like "Decrto" or "Portatia" without false positives.

   **Behavior rules:**
   - High confidence: Auto-populate structured filter chips as *suggestions* (not applied ŌĆö consistent with Blocker 2 fix). If exact document match found, offer "Ir para documento" action alongside results.
   - Medium confidence: Show disambiguation UI ŌĆö "Voc├¬ quis dizer: [option A] ou [option B]?" with structured filter chips for each interpretation.
   - Low confidence: Pass through as keyword search unchanged. No auto-parsing.
   - Parse failure: Fall back silently to full-text keyword search. Never block the user's query.
   - **Never auto-navigate on parse alone.** Always show results list even on exact match ŌĆö the user may be looking for amendments, related acts, or context, not the act itself.

   **Threshold validation requirement:** Confidence thresholds must not ship with arbitrary values. Before production deployment: (a) collect a corpus of at least 1,000 real search queries from DOU (from access logs, user research, or synthetic generation from actual DOU document headers), (b) measure parse precision and recall against this corpus, (c) set thresholds to achieve Ōēź95% precision (false positives are more dangerous than false negatives for legal search). Ship initially with the most conservative behavior: only parse unambiguous formats (`Lei X.XXX/YYYY`, `Decreto-Lei X.XXX/YYYY`) and widen pattern acceptance based on measured accuracy data from production query logs.

2. **Boolean query support.** Professional search requires AND/OR/NOT operators with grouping. Without this, complex compliance queries ("all INs from ANVISA referencing RDC 330 OR RDC 331 excluding corrections") are impossible.

   **Syntax recommendation:** Adopt Lucene query syntax (`AND`, `OR`, `NOT`, `"exact phrase"`, parenthetical grouping). Legal professionals migrating from Westlaw or LexisNexis already have mental models for boolean search. Lucene syntax is directly passable to Elasticsearch/OpenSearch backends without translation. Support Portuguese aliases (`E`/`OU`/`N├āO`) alongside English operators to accommodate local usage patterns.

   **Syntax validation and error feedback:** If a user types malformed boolean syntax (unclosed parentheses, orphaned operators, adjacent operators), the system must show inline syntax error highlighting with a correction suggestion (e.g., `"RDC 330 OR" ŌåÆ Operador "OR" sem termo seguinte`). Do not silently fall back to keyword search for malformed boolean queries ŌĆö this masks errors and produces misleading result sets. Only fall back to keyword search for queries that contain no boolean operators at all.

   **Alternative input:** Provide a visual query builder (structured form with AND/OR/NOT dropdowns and field selectors) as an alternative to raw syntax for users who prefer guided construction. Both modes must produce the same underlying query representation.

3. **Saved searches and monitoring.** Every search is ephemeral. Audit engagements run weeks to months. Compliance monitoring is ongoing. Without saved queries with optional change alerts, the tool is useful for spot lookups but not sustained investigation.

### Normative Awareness: Present but Not Trustworthy

As detailed in Blocker 1, the normative awareness layer has the right *shape* (status card, timeline, reference lists) but the wrong *confidence model*. The information architecture is sound; the implementation is unsound because it doesn't distinguish between verified and inferred status.

**What's missing for investigation-grade normative awareness:**

1. **Point-in-time document reconstruction.** "What was Article 7 on March 15, 2022?" is a fundamental audit question. Without temporal versioning (valid_from/valid_to ranges on text segments, date picker in reader), historical compliance analysis is impossible. This is a backend-dependent feature ŌĆö see Section 5, Priority 5 for phased approach.

2. **Bidirectional citation navigation.** "Cites" and "Cited by" with traversable links. The current normative_refs and procedure_refs are unidirectional (outbound only). Legal reasoning requires knowing what *later* acts reference or modify the current document.

3. **Data provenance on every metadata element.** Source URL, scrape timestamp, content checksum, last verification date. This must be visible (not hidden behind settings) in an expandable "Trilha de auditoria" section on every document. Professional users must be able to demonstrate to courts and regulators that their data source is reliable and current. **Accessibility requirement:** The expandable provenance section must be keyboard-navigable (Enter/Space to toggle, arrow keys within), announce expansion state to screen readers via `aria-expanded`, and meet WCAG 2.1 AA contrast ratios for all text including timestamps and hash values.

## 5. Recommended Next Steps, Prioritized by Impact

### Priority 0 ŌĆö Content Security Hardening (1 week, non-negotiable before any user exposure)
1. Implement server-side HTML sanitization at ingest with element/attribute whitelist. Convert inline styles to CSS utility classes during sanitization.
2. Add client-side DOMPurify in DocumentReader component with matching whitelist config.
3. Deploy CSP headers **without `unsafe-inline`** for any directive. Add SRI to all static assets and all external dependencies (fonts, analytics, any CDN-loaded resource).
4. Implement automated XSS regression test suite (OWASP XSS Filter Evasion patterns) as a CI gate on ingest pipeline changes.
5. **Risk:** Low. Well-understood problem with mature tooling. **Rollback:** If class-mapping misses edge-case DOU inline styles, implement hash-based CSP for those specific styles (not blanket `unsafe-inline`). Debug whitelist against real DOU corpus.
6. **Dependency:** None. Can proceed immediately.

### Priority 1 ŌĆö Fix Trust Blockers + Accessibility Minimum Viable Gate (2 weeks)

**1a. Normative status ŌĆö Phase A (production gate):** Remove the ValidityCard from rendering for all heuristic data. Replace with footer link to official Planalto source. Build the ValidityCard component (with `verified` rendering path) but gate display on `metadata_quality: 'authoritative'` from the API. Add residual risk disclosure to product terms of use. **Do not promise the `verified` tier on near-term roadmaps** ŌĆö it requires a 3ŌĆō6 month backend pipeline (SILEG/LexML integration, conflict resolution, retroactive modification handling). **Risk:** Users may perceive the tool as less capable than competitors that show status without caveats. This is the correct tradeoff ŌĆö false confidence is worse than honest uncertainty for the target audience. **Dependency:** Backend must add `metadata_quality` field to API response.

**1b. Inferred filters as suggestions, not constraints:** Refactor FilterPanel to present inferred filters as dismissible suggestions in a "Sistema sugeriu" chip group. Implement query audit log as local-only IndexedDB storage (no encryption, no server transmission). Export function with full/methodology-only choice. 30-day auto-expiry. **Risk:** Low ŌĆö straightforward frontend work. **Dependency:** None.

**1c. Search state preservation + document route context bar:** Implement URL-based search state preservation using `replaceState` for filter changes and `pushState` for new searches. When serialized state exceeds 1.5KB, fall back to sessionStorage with UUID reference in URL. Add opt-in context bar component that document routes can render. Define tablet breakpoint behavior (collapsed nav rail at 768pxŌĆō1024px). **Risk:** Low. **Dependency:** None.

**1d. Accessibility minimum viable gate (must pass before production):** The following accessibility criteria are production blockers, not deferred improvements: (a) keyboard navigation for all interactive elements (filters, chips, document TOC, expandable sections), (b) screen reader support for document TOC navigation (article hierarchy must be navigable), (c) color contrast Ōēź 4.5:1 for all text including footer links and status indicators, (d) `aria-expanded` on all expandable sections, `aria-live` on search result count updates. Run axe-core automated audit; fix all Critical and Serious violations before production. Manual screen reader testing (NVDA + Chrome, VoiceOver + Safari) for document reading workflow. **Risk:** Accessibility retrofitting cost increases with each iteration that defers it. Starting now minimizes total cost. **Dependency:** None.

### Priority 2 ŌĆö Performance Baseline + Large Document Handling (2 weeks, parallel with Priority 1)
1. **Establish baselines:** Instrument and measure all "Not measured" items in the trajectory table. This is diagnostic work, not optimization ŌĆö you cannot improve what you haven't measured.
2. **Performance budget:** Set bundle size cap (200KB gzipped JS for initial load). Implement virtualized scrolling in DocumentReader for documents exceeding 500KB rendered HTML. Add pagination with configurable page size (20/50/100) for search results. DOU documents can reach extreme sizes; the reader must not crash or freeze the browser tab.
3. **Risk:** Low. **Dependency:** None.

### Priority 3 ŌĆö Citation Parsing (2ŌĆō3 weeks, highest single-feature ROI)
Implement citation parser in the search bar per the expanded grammar and confidence rules in Section 3. High-confidence parses populate suggested filter chips (not auto-applied). Medium-confidence parses show disambiguation UI. Low-confidence and failures fall through to keyword search. Fuzzy type matching via Levenshtein distance for typo tolerance.

**Ship with conservative thresholds:** Initially parse only unambiguous formats (`Lei X.XXX/YYYY`, `Decreto-Lei X.XXX/YYYY`, full-format citations with "n┬║"/"n."). Validate against a corpus of Ōēź1,000 real DOU search queries before widening pattern acceptance. Target Ōēź95% precision before expanding to medium-confidence patterns.

**Never auto-navigate; always show results.** **Risk:** False positive parses are worse than no parser. The corpus validation requirement and conservative initial thresholds mitigate this. Log all parse attempts with outcomes to enable data-driven threshold tuning. **Dependency:** None for basic parsing. Exact-match document lookup requires API endpoint.

### Priority 4 ŌĆö Data Provenance Layer (1ŌĆō2 weeks)
Add expandable "Trilha de auditoria" section to every document page showing: source URL, scrape/sync timestamp, content hash (SHA-256), last verification date, ingest pipeline version. Expose in API types and render in reader. Keyboard-navigable, screen-reader-announced (`aria-expanded`). Low effort, high trust signal for professional users. **Risk:** Low. **Dependency:** Backend must expose provenance metadata in document API response.

### Priority 5 ŌĆö Temporal Versioning (Backend-dependent, timeline TBD)
