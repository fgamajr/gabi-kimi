# Frontend Audit: DOU Research Tool — Iteration Review

## Executive Summary

This iteration delivers real structural progress — shared app shell, query transparency, and a first normative-awareness layer — but the product remains a **document browser with investigation aspirations**, not yet an investigation tool. Three of the implemented changes carry design decisions that are actively dangerous for the target user base: heuristic-based legal validity without confidence signaling, auto-applied inferred search filters, and unaddressed content sanitization in the document reader. These must be corrected before any production exposure to legal/audit professionals.

---

## 1. What Actually Improved (Capability Gains, Not Cosmetic)

### Shared App Shell + Route-Level Layout
**Real gain.** Persistent navigation rail (desktop) and bottom nav (mobile) establish consistent wayfinding across search and reading workflows. Route-level layout control is architecturally sound — it allows document routes to manage their own chrome without forking the shell. This is infrastructure that compounds. However, as noted in Blocker 3, this architectural choice creates a tension with investigation-loop navigation that must be resolved explicitly rather than papered over with shell-imposed chrome.

### Query Transparency via interpreted_query / inferred_filters / applied_filters
**Real gain.** Surfacing the system's interpretation of a query is a prerequisite for professional trust. Legal researchers must know *exactly* what the system searched for. The separation into three distinct layers (raw interpretation, system inference, user-confirmed filters) is the correct data model. However, the **presentation** of this model has a critical flaw (see Section 2, Blocker 2).

### Normative Timeline Derived from References
**Partial gain.** Showing a timeline of connected normative references is useful orientation. But the value is capped by the heuristic validity layer it depends on — a timeline built on uncertain data is a timeline that cannot be cited.

## 2. What Still Blocks Production Usability

### BLOCKER 1 — Heuristic Normative Status Is a Liability, Not a Feature

**Severity: Critical. Both reviewers flagged this independently. They are correct.**

The "Situação normativa" card uses "cautious heuristics" to infer revocation, replacement, alteration, and complementation signals. For a general-audience news reader, this might be acceptable with caveats. For legal and audit professionals — the stated target — this is worse than showing nothing.

**Why it's dangerous:** A legal professional who sees a status card that says "revogado" (revoked) or shows no revocation signal will treat that as authoritative unless the UI aggressively communicates otherwise. A false negative (showing a revoked law as active) can lead to reliance on invalid legislation. A false positive (showing a valid law as revoked) can cause a researcher to miss controlling authority.

**Required correction — phased approach acknowledging backend dependency:**

**Phase A — Immediate (1–2 weeks, frontend only):** Until an authoritative data pipeline exists, *all* normative status data is heuristic. The ValidityCard must treat everything as unverified and render accordingly. No "Verified" tier until the backend can deliver it.

**Phase B — Epic (3–6 months, backend pipeline):** Integration with Planalto's consolidation database (SILEG) and LexML, with conflict resolution, retroactive modification handling, and temporal graph maintenance. Only after this pipeline is operational does the "Verified" tier become available to the frontend.

**ValidityCard component specification:**

```typescript
type ConfidenceLevel = 'verified' | 'unverified' | 'unknown';

interface ProvenanceInfo {
  sourceUrl: string;          // e.g., Planalto URL, LexML URN
  sourceLabel: string;        // e.g., "Planalto — Legislação Consolidada"
  syncTimestamp: string;      // ISO 8601
  contentHash: string;        // SHA-256 of source document at sync time
  verificationMethod: string; // e.g., "sileg_api" | "lexml_crossref" | "dou_heuristic"
}

interface ValidityCardProps {
  confidenceLevel: ConfidenceLevel;
  status: string | null;              // "vigente" | "revogado" | "alterado" | null
  signals: NormativeSignal[];         // parsed heuristic signals (revogação, alteração, etc.)
  provenance: ProvenanceInfo | null;
  officialSourceUrl: string;          // always present — link to authoritative source
}
```

**Rendering rules per confidence level:**

| Level | Banner | Status display | Provenance footer |
|---|---|---|---|
| `verified` | Green — "Verificado em [source] em [date]" | Displayed as determined status | Full provenance: source, sync timestamp, hash, method |
| `unverified` | Amber — "Situação inferida por heurística — conferir na fonte oficial" | Displayed as *leads*, not determinations. Copy: "Sinais identificados (não verificados):" followed by signal list | Partial provenance: method = "dou_heuristic", sync timestamp, link to official source |
| `unknown` | Neutral — "Situação normativa não disponível" | No status displayed | Link to official consolidated source only |

**Phase A ships with `confidenceLevel` hardcoded to `'unverified'` for any document with heuristic signals, and `'unknown'` for documents without.** The `'verified'` path is implemented in the component but gated on a backend capability flag (`metadata_quality: 'authoritative'` in the API response). This prevents frontend work from being wasted while honestly representing current data quality.

**Backend API contract required:**

```typescript
interface NormativeMetadataResponse {
  metadata_quality: 'authoritative' | 'heuristic' | 'none';
  status: string | null;
  signals: NormativeSignal[];
  provenance: ProvenanceInfo | null;
  official_source_url: string;
}
```

### BLOCKER 2 — Inferred Filters Auto-Applied Without User Confirmation

**Severity: High.**

The search interface currently surfaces `inferred_filters` as applied constraints alongside user-selected `applied_filters`. In legal discovery and audit workflows, the distinction between "what I asked for" and "what the system decided I meant" is the difference between a defensible search methodology and a negligent one.

**Why it's dangerous:** If a system silently narrows a search — for example, interpreting "decreto 2018" as `type:decreto AND year:2018` when the user intended a keyword search across all document types mentioning "decreto 2018" — it can exclude relevant results. The user cannot verify completeness of their research if the system is silently shaping their query.

**Required correction:**
- `inferred_filters` must be surfaced as **suggested filter chips** in a visually distinct "Sistema sugeriu" group, not auto-applied to the query.
- User must explicitly accept each inferred filter (tap/click to promote to `applied_filters`) or dismiss it.
- Once accepted, the filter moves to the standard applied filter chip group with no visual distinction — it's now the user's filter.
- The `interpreted_query` display should remain visible as-is (showing raw system interpretation is good transparency), but it must be clearly labeled as diagnostic, not authoritative.

**Query Audit Log — with privacy controls:**

Every search session should be exportable as a structured record for legal research reproducibility. However, legal research queries often contain privileged attorney-client communications, litigation strategy, or sensitive investigation targets. Logging these without safeguards creates malpractice exposure and subpoena risk.

```typescript
interface QueryAuditRecord {
  sessionId: string;                  // UUID, generated client-side
  timestamp: string;                  // ISO 8601
  methodology: {                      // safe to retain — describes process, not content
    inferredFilters: FilterSpec[];
    userDecisions: { filter: FilterSpec; action: 'accepted' | 'rejected' }[];
    appliedFilters: FilterSpec[];
    resultCount: number;
  };
  content: {                          // sensitive — subject to encryption + expiry
    rawQuery: string;
    interpretedQuery: string;
  };
}

interface QueryAuditExport {
  format: 'json' | 'csv' | 'pdf';
  records: QueryAuditRecord[];
  exportTimestamp: string;
  toolVersion: string;
}
```

**Privacy requirements:**
- The `content` block must be encrypted client-side with a user-controlled key before any persistence (localStorage, IndexedDB, or server). The `methodology` block may be stored unencrypted — it describes search logic, not search intent.
- Auto-expiry: 30 days by default, user-configurable per session. No server-side retention of raw query text without explicit user consent.
- Explicit consent flow: on first export attempt, prompt user to acknowledge that the audit log will contain their search terms and offer methodology-only export as an alternative.
- LGPD compliance: if any server-side storage is added later, it must include data subject access and deletion endpoints.

### BLOCKER 3 — Mobile Navigation on Document Routes Breaks Investigation Loop

**Severity: Medium, but high-friction for the core workflow.**

Legal research is not linear reading — it's a rapid oscillation between search results and document inspection. Hiding mobile bottom nav on document routes forces reliance on browser back-button, which destroys filter state and scroll position in the results list.

**Architectural clarification:** The route-level layout autonomy praised in Section 1 is the right architecture. The solution must not contradict it by forcing shell-imposed chrome onto document routes. Instead, the navigation problem should be solved at the state management layer.

**Required correction — two complementary measures:**

1. **Search state preservation (mandatory, solves the root cause):** Search state (query, filters, scroll position, result set page) must be preserved in URL search parameters and/or sessionStorage so that browser back-button, swipe-back, or any navigation to the search route restores the exact prior state. Without this, no amount of navigation chrome fixes the underlying problem. This is the highest-priority fix.

2. **Document route opt-in navigation (respects route autonomy):** Document routes *choose* to render a lightweight context bar (not imposed by the shell). This bar is part of the document route's own layout, showing: [← Resultados (N)] | [Filter badge count]. The document route requests this from a shared component; the shell does not inject it. This preserves the architectural principle while providing wayfinding.

3. **Tablet breakpoint (768px–1024px):** Define explicit behavior — tablet uses the desktop navigation rail in collapsed (icon-only) mode. Do not leave this breakpoint undefined; it's a common device class for legal professionals using iPads in courtrooms and audit rooms.

### BLOCKER 4 — Unsanitized Document Rendering Creates XSS Vectors

**Severity: Critical.**

The document reader renders HTML content scraped from DOU. DOU documents contain complex HTML tables, inline styles, embedded metadata, and potentially malformed markup. A compromised scrape pipeline, a malicious document injection, or even a legitimate DOU publication with unusual markup could execute scripts in the context of a legal research session, exposing privileged client data, session tokens, or enabling session hijacking.

**Why it's critical for this specific product:** The target users — attorneys and auditors — handle privileged information. An XSS vector in a document reader they trust becomes an attack surface against attorney-client privilege. This is not a theoretical concern; government gazette scrapers are high-value targets for supply chain attacks.

**Required correction:**

1. **Server-side sanitization at ingest time (defense layer 1):** All scraped HTML must be sanitized through a whitelist-based sanitizer before storage. Allowed elements: `table`, `thead`, `tbody`, `tr`, `th`, `td`, `p`, `br`, `strong`, `em`, `b`, `i`, `u`, `sup`, `sub`, `ol`, `ul`, `li`, `h1`–`h6`, `span`, `div`, `caption`, `colgroup`, `col`. Allowed attributes: `colspan`, `rowspan`, `class`, `id`. Strip all `style` attributes, `on*` event handlers, `href="javascript:..."`, `<script>`, `<iframe>`, `<object>`, `<embed>`, `<form>`, and `data-*` attributes. Store both raw (for re-processing) and sanitized (for serving) versions.

2. **Client-side DOMPurify (defense layer 2):** Before rendering any document HTML in the reader component, pass through DOMPurify with a restrictive config matching the server-side whitelist. This is defense-in-depth against server-side sanitizer bypass or pipeline compromise.

3. **Content Security Policy headers:**
   ```
   Content-Security-Policy:
     default-src 'self';
     script-src 'self';
     style-src 'self' 'unsafe-inline';
     img-src 'self' data: https://www.in.gov.br;
     frame-ancestors 'none';
     base-uri 'self';
   ```
   Note: `'unsafe-inline'` for styles only — DOU tables rely on inline styles for column widths. `script-src` must NOT include `'unsafe-inline'` or `'unsafe-eval'`.

4. **Subresource integrity (SRI)** for all static JS/CSS assets served from CDN.

## 3. Depth Assessment: Are Search Filters and Normative Awareness Investigation-Grade?

### Search Filters: Structured but Shallow

The current filter set (section, act type, issuing organ, date range) covers the basic facets of DOU document metadata. This is necessary but insufficient for professional legal search.

**What's missing for investigation-grade search:**

1. **Citation parsing in the search bar.** Legal professionals navigate by citation, not by keyword. This is the single highest-impact search UX improvement for the target audience. Current dropdown-based decomposition of citations creates unacceptable friction.

   **Citation parser specification:**

   Brazilian legal citations are highly variable. The parser must handle these formats with explicit confidence thresholds:

   | Format | Example | Confidence |
   |---|---|---|
   | Compact with year | `Lei 13.709/2018` | High — parse to type + number + year |
   | With "nº" | `Lei nº 13.709, de 14 de agosto de 2018` | High — parse to type + number + full date |
   | Abbreviated type | `IN RFB 1.234/2020` | Medium — "IN" is ambiguous (Instrução Normativa from multiple organs) |
   | Popular name | `LGPD` | Low — requires lookup table, not parsing |
   | Partial | `Decreto 2018` | Ambiguous — could be decree #2018 or year filter |

   **Behavior rules:**
   - High confidence (≥0.9): Auto-populate structured filter chips as *suggestions* (not applied — consistent with Blocker 2 fix). If exact document match found, offer "Ir para documento" action.
   - Medium confidence (0.6–0.9): Show disambiguation UI — "Você quis dizer: [option A] ou [option B]?" with structured filter chips for each interpretation.
   - Low confidence (<0.6): Pass through as keyword search unchanged. No auto-parsing.
   - Parse failure: Fall back silently to full-text keyword search. Never block the user's query.
   - **Never auto-navigate on parse alone.** Always show results list even on exact match — the user may be looking for amendments, related acts, or context, not the act itself.

   Grammar (simplified PEG notation for implementation guidance):
   ```
   Citation     ← TypeToken Space? "nº"? Space? NumberToken (Space? "/" Space? YearToken)?
   TypeToken    ← "Lei" / "Decreto" / "Portaria" / "IN" / "RDC" / "Resolução" / ...
   NumberToken  ← [0-9] ("." [0-9]{3})* / [0-9]+
   YearToken    ← [0-9]{4}
   ```

2. **Boolean query support.** Professional search requires AND/OR/NOT operators with grouping. Without this, complex compliance queries ("all INs from ANVISA referencing RDC 330 OR RDC 331 excluding corrections") are impossible.

   **Syntax recommendation:** Adopt Lucene query syntax. Legal professionals migrating from Westlaw or LexisNexis already have mental models for boolean search. Lucene syntax (`AND`, `OR`, `NOT`, `"exact phrase"`, parenthetical grouping) is the closest match and has the advantage of being directly passable to Elasticsearch/OpenSearch backends without translation. Custom syntax is a trap — it requires documentation, training, and long-term maintenance for no user benefit.

3. **Saved searches and monitoring.** Every search is ephemeral. Audit engagements run weeks to months. Compliance monitoring is ongoing. Without saved queries with optional change alerts, the tool is useful for spot lookups but not sustained investigation.

### Normative Awareness: Present but Not Trustworthy

As detailed in Blocker 1, the normative awareness layer has the right *shape* (status card, timeline, reference lists) but the wrong *confidence model*. The information architecture is sound; the implementation is unsound because it doesn't distinguish between verified and inferred status.

**What's missing for investigation-grade normative awareness:**

1. **Point-in-time document reconstruction.** "What was Article 7 on March 15, 2022?" is a fundamental audit question. Without temporal versioning (valid_from/valid_to ranges on text segments, date picker in reader), historical compliance analysis is impossible. This is a backend-dependent feature — see Section 5, Priority 4 for phased approach.

2. **Bidirectional citation navigation.** "Cites" and "Cited by" with traversable links. The current normative_refs and procedure_refs are unidirectional (outbound only). Legal reasoning requires knowing what *later* acts reference or modify the current document.

3. **Data provenance on every metadata element.** Source URL, scrape timestamp, content checksum, last verification date. This must be visible (not hidden behind settings) in an expandable "Trilha de auditoria" section on every document. Professional users must be able to demonstrate to courts and regulators that their data source is reliable and current. **Accessibility requirement:** The expandable provenance section must be keyboard-navigable (Enter/Space to toggle, arrow keys within), announce expansion state to screen readers via `aria-expanded`, and meet WCAG 2.1 AA contrast ratios for all text including timestamps and hash values.

## 4. Overall Trajectory Assessment

The product is **moving in the right direction** but is not yet on the investigation-grade track.

| Capability Layer | Current State | Investigation-Grade Threshold | Performance Target |
|---|---|---|---|
| Navigation / Shell | ✅ Solid | Adequate | Route transitions < 200ms |
| Document Reading | ✅ Strong (TOC, progress, deep-link, export) | Adequate with minor gaps | Document render < 500ms for 95th percentile document size |
| Content Security | ❌ No visible sanitization | Requires server + client sanitization + CSP | N/A — binary pass/fail |
| Normative Awareness | ⚠️ Present but untrustworthy | Requires confidence model + provenance | Provenance data load < 200ms |
| Search Structure | ⚠️ Basic facets, no citation parsing | Requires citation parser + Boolean + saved searches | Filter response < 100ms, search results < 1s |
| Query Transparency | ⚠️ Good data model, bad UX (auto-apply) | Requires user confirmation of inferred filters | Suggestion render < 50ms |
| Temporal Analysis | ❌ Absent | Requires point-in-time reconstruction | Backend-dependent |
| Workspace / Case Management | ❌ Absent | Requires saved searches + folders + alerts | Deferred |
| Data Provenance / Audit Trail | ❌ Absent | Requires visible lineage on all metadata | Footer render with document |
| Accessibility (WCAG 2.1 AA) | ❌ Not assessed | Screen reader support, keyboard nav, contrast compliance | Mandatory for gov/law firm procurement |
| Performance / Large Documents | ❌ Not assessed | Virtualized scrolling, bundle budget, streaming results | First paint < 3s on 3G; 50MB doc renders without crash |

The iteration improved the **floor** (consistent navigation, basic transparency) without raising the **ceiling** (trustworthy legal data, professional workflow support). The product currently serves a "legal-curious generalist." It does not yet serve the "investigating auditor" or "compliance attorney" who needs to build a defensible research record.

## 5. Recommended Next Steps, Prioritized by Impact

### Priority 0 — Content Security Hardening (1 week, non-negotiable before any user exposure)
1. Implement server-side HTML sanitization at ingest with element/attribute whitelist.
2. Add client-side DOMPurify in DocumentReader component with matching whitelist config.
3. Deploy CSP headers. Add SRI to all static assets.
4. **Risk:** Low. Well-understood problem with mature tooling. **Rollback:** Revert CSP if legitimate DOU content is blocked; debug whitelist.
5. **Dependency:** None. Can proceed immediately.

### Priority 1 — Fix Trust Blockers (1–2 weeks)
1. **Normative status — Phase A only:** Implement ValidityCard with `unverified`/`unknown` modes. Hardcode all heuristic data to `unverified` with amber banner. Add provenance footer with sync timestamp and official source link. **Do not promise the `verified` tier** — it requires a 3–6 month backend pipeline (SILEG/LexML integration, conflict resolution, retroactive modification handling). Design the component to accept `verified` when the backend delivers it; do not build frontend around data that doesn't exist yet. **Risk:** Users may perceive the tool as less capable than competitors that show status without caveats. This is the correct tradeoff — false confidence is worse than honest uncertainty for the target audience. **Dependency:** Backend must add `metadata_quality` field to API response.

2. **Inferred filters as suggestions, not constraints:** Refactor FilterPanel to present inferred filters as dismissible suggestions in a "Sistema sugeriu" chip group. Implement query audit log with privacy controls (client-side encryption for content block, 30-day auto-expiry, methodology/content separation, consent flow). **Risk:** Medium — encryption key management UX is non-trivial. Start with local-only export (no server storage) to defer the hard problems. **Dependency:** None — frontend only.

3. **Search state preservation + document route context bar:** Implement URL-based search state preservation first (this is the root cause fix). Then add opt-in context bar component that document routes can render. Define tablet breakpoint behavior (collapsed nav rail). **Risk:** Low. **Dependency:** None.

### Priority 2 — Content Security + Accessibility Baseline (2 weeks, parallel with Priority 1)
1. **Performance budget:** Set bundle size cap (200KB gzipped JS for initial load). Implement virtualized scrolling in DocumentReader for documents exceeding 500KB rendered HTML. Add pagination with configurable page size (20/50/100) for search results. DOU documents can reach extreme sizes; the reader must not crash or freeze the browser tab.
2. **WCAG 2.1 AA audit:** Screen reader testing for document TOC navigation (article hierarchy: art. 5º → § 2º → inciso I must be navigable), keyboard navigation for all interactive elements, color contrast verification for confidence indicators (amber unverified banner must meet 4.5:1 ratio against background). This is a procurement blocker for government agencies and large law firms.
3. **Risk:** Accessibility retrofitting is more expensive than accessible-first development. Flag this as increasing cost with each iteration that defers it. **Dependency:** None.

### Priority 3 — Citation Parsing (2–3 weeks, highest single-feature ROI)
Implement citation parser in the search bar per the grammar and confidence rules in Section 3. High-confidence parses populate suggested filter chips (not auto-applied). Medium-confidence parses show disambiguation UI. Low-confidence and failures fall through to keyword search. **Never auto-navigate; always show results.** **Risk:** False positive parses are worse than no parser — a user who types "Decreto 9.492" and gets auto-filtered to decree type + number 9492 when they wanted a keyword search will miss results. The confidence threshold system mitigates this, but requires real-world query logs to tune. Ship with conservative thresholds (only parse high-confidence patterns) and widen based on data. **Dependency:** None for basic parsing. Exact-match document lookup requires API endpoint.

### Priority 4 — Data Provenance Layer (1–2 weeks)
Add expandable "Trilha de auditoria" section to every document page showing: source URL, scrape/sync timestamp, content hash (SHA-256), last verification date, ingest pipeline version. Expose in API types and render in reader. Keyboard-navigable, screen-reader-announced. Low effort, high trust signal for professional users. **Risk:** Low. **Dependency:** Backend must expose provenance metadata in document API response.

### Priority 5 — Temporal Versioning (Backend-dependent, timeline TBD)
**This cannot be responsibly estimated without a backend architecture review.** Frontend design for point-in-time reconstruction (date picker in reader header, version diff view, temporal query parameter) requires knowing: (a) whether the backend stores historical document versions or only current state, (b) what granularity of versioning is available (full document vs. article-level), (c) query performance characteristics of temporal lookups, (d) storage cost model for version proliferation.

**Recommended approach:**
- **Immediate (1 week):** Conduct backend architecture review. Document current storage model, identify gaps for temporal versioning, estimate pipeline work.
- **Phase 1 (frontend, 2 weeks, after backend review):** Design the date picker and diff view UI as interactive mockups. Validate with target users before committing to implementation.
- **Phase 2 (backend + frontend, 4–12 weeks depending on review findings):** Implement temporal storage, version indexing, and frontend integration.
- **Rollback:** If backend review reveals temporal versioning requires a storage architecture migration, re-scope to "document change log" (list of known modifications with dates and links to modifying acts) as a lower-cost alternative that still provides significant audit value.

### Priority 6 — Workspace / Case Management (4–6 weeks)
Saved searches, case folders, annotation/bookmarking within documents, search alert subscriptions. Requires authentication infrastructure. Defer implementation but begin data model design now — localStorage-based recent history is a stepping stone, not a destination. **Risk:** Auth infrastructure is a project in itself. Don't let workspace features become the forcing function for auth — ship auth as its own priority when the user base justifies it. **Dependency:** Authentication system, notification infrastructure for alerts.

## Final Verdict

**Not yet production-ready for professional legal/audit users.** The four blockers — content security, untrustworthy normative status presentation, auto-applied inferred filters, and broken investigation-loop navigation — must be resolved before any professional user exposure. Content security (P0) is the hardest gate: rendering unsanitized third-party HTML to attorneys handling privileged matters is an unacceptable risk regardless of feature completeness.

After those fixes, citation parsing is the single feature that most changes the product's competitive position. The architecture is heading in the right direction; the confidence model, security posture, accessibility baseline, and workflow support need to catch up to the information architecture ambitions.

The normative status "Verified" tier is a 3–6 month backend investment, not a frontend sprint item. Do not promise it on near-term roadmaps. The honest "Não verificado" label, properly implemented, is more valuable to the target audience than a false "Verificado" built on heuristics.
