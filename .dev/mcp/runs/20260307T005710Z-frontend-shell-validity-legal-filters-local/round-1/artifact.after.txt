# Frontend Audit: DOU Research Tool — Iteration Review

## Executive Summary

This iteration delivers real structural progress — shared app shell, query transparency, and a first normative-awareness layer — but the product remains a **document browser with investigation aspirations**, not yet an investigation tool. Two of the three implemented changes carry design decisions that are actively dangerous for the target user base: heuristic-based legal validity and auto-applied inferred search filters. These must be corrected before any production exposure to legal/audit professionals.

---

## 1. What Actually Improved (Capability Gains, Not Cosmetic)

### Shared App Shell + Route-Level Layout
**Real gain.** Persistent navigation rail (desktop) and bottom nav (mobile) establish consistent wayfinding across search and reading workflows. Route-level layout control is architecturally sound — it allows document routes to manage their own chrome without forking the shell. This is infrastructure that compounds.

### Query Transparency via interpreted_query / inferred_filters / applied_filters
**Real gain.** Surfacing the system's interpretation of a query is a prerequisite for professional trust. Legal researchers must know *exactly* what the system searched for. The separation into three distinct layers (raw interpretation, system inference, user-confirmed filters) is the correct data model. However, the **presentation** of this model has a critical flaw (see Section 3).

### Normative Timeline Derived from References
**Partial gain.** Showing a timeline of connected normative references is useful orientation. But the value is capped by the heuristic validity layer it depends on — a timeline built on uncertain data is a timeline that cannot be cited.

## 2. What Still Blocks Production Usability

### BLOCKER 1 — Heuristic Normative Status Is a Liability, Not a Feature

**Severity: Critical. Both reviewers flagged this independently. They are correct.**

The "Situação normativa" card uses "cautious heuristics" to infer revocation, replacement, alteration, and complementation signals. For a general-audience news reader, this might be acceptable with caveats. For legal and audit professionals — the stated target — this is worse than showing nothing.

**Why it's dangerous:** A legal professional who sees a status card that says "revogado" (revoked) or shows no revocation signal will treat that as authoritative unless the UI aggressively communicates otherwise. "Cautious heuristics" means the system will sometimes be wrong. In legal research, a false negative (showing a revoked law as active) can lead to reliance on invalid legislation. A false positive (showing a valid law as revoked) can cause a researcher to miss controlling authority.

**Required correction:**
- The validity card must operate in **three explicit modes**, driven by backend metadata quality:
  - **Verified**: Backend provides explicit, linked revocation/amendment metadata with source provenance (e.g., a specific revoking act ID from Planalto/LexML). Display status with green confidence indicator and data lineage (source, sync timestamp).
  - **Unverified / Heuristic**: Backend provides only parsed signals without authoritative confirmation. Display with amber "Não verificado — conferir na fonte oficial" banner. Do not present as a status determination. Present as *leads for manual verification*.
  - **Unknown**: No structured metadata available. Display explicit "Situação normativa não disponível" with direct link to official consolidated source.
- Every normative status display must include a **provenance footer**: data source, last synchronization timestamp, and a link to the authoritative source document. This is non-negotiable for audit trail requirements.
- The heuristic logic can remain as a **research aid** — surfacing signals for the user to verify — but it must never be presented in the same visual language as deterministic status.

### BLOCKER 2 — Inferred Filters Auto-Applied Without User Confirmation

**Severity: High.**

The search interface currently surfaces `inferred_filters` as applied constraints alongside user-selected `applied_filters`. In legal discovery and audit workflows, the distinction between "what I asked for" and "what the system decided I meant" is the difference between a defensible search methodology and a negligent one.

**Why it's dangerous:** If a system silently narrows a search based on NLP inference — for example, interpreting "decreto 2018" as `type:decreto AND year:2018` when the user intended a keyword search across all document types mentioning "decreto 2018" — it can exclude relevant results. The user cannot verify completeness of their research if the system is silently shaping their query.

**Required correction:**
- `inferred_filters` must be surfaced as **suggested filter chips** in a visually distinct "Sistema sugeriu" group, not auto-applied to the query.
- User must explicitly accept each inferred filter (tap/click to promote to `applied_filters`) or dismiss it.
- Once accepted, the filter moves to the standard applied filter chip group with no visual distinction — it's now the user's filter.
- The `interpreted_query` display should remain visible as-is (showing raw system interpretation is good transparency), but it must be clearly labeled as diagnostic, not authoritative.
- Implement a **Query Audit Log** export: every search session should be exportable as a structured record (timestamp, raw query, inferred filters offered, filters accepted/rejected, applied filters, result count). This is table stakes for legal research reproducibility.

### BLOCKER 3 — Mobile Navigation Hidden on Document Routes Breaks Investigation Loop

**Severity: Medium, but high-friction for the core workflow.**

Legal research is not linear reading — it's a rapid oscillation between search results and document inspection. Hiding mobile bottom nav on document routes forces reliance on browser back-button, which destroys filter state and scroll position in the results list.

**Required correction:**
- Option A (recommended): Retain bottom nav on document routes but collapse it to a single-row **context bar** showing: [← Search results (N)] | [filters badge] | [Home]. This preserves wayfinding without consuming the vertical space needed for document reading.
- Option B: Keep bottom nav fully visible on all routes. Less elegant but functional.
- In either case: search state (query, filters, scroll position, result set) must be preserved in session/URL state so that returning to search results is instantaneous and lossless. Without this, no amount of navigation chrome fixes the underlying problem.

## 3. Depth Assessment: Are Search Filters and Normative Awareness Investigation-Grade?

### Search Filters: Structured but Shallow

The current filter set (section, art type, issuing organ, date range) covers the basic facets of DOU document metadata. This is necessary but insufficient for professional legal search.

**What's missing for investigation-grade search:**
1. **Citation parsing in the search bar.** Legal professionals navigate by citation, not by keyword. Typing "Lei 13.709/2018, art. 7º, §3º" should auto-parse into structured filters (type: Lei, number: 13709, year: 2018) and, on match, jump directly to the article-level anchor. This is the single highest-impact search UX improvement for the target audience. Current dropdown-based decomposition of citations creates unacceptable friction.
2. **Boolean query support.** Professional search requires AND/OR/NOT operators with grouping. Without this, complex compliance queries ("all INs from ANVISA referencing RDC 330 OR RDC 331 excluding corrections") are impossible.
3. **Saved searches and monitoring.** Every search is ephemeral. Audit engagements run weeks to months. Compliance monitoring is ongoing. Without saved queries with optional change alerts, the tool is useful for spot lookups but not sustained investigation.

### Normative Awareness: Present but Not Trustworthy

As detailed in Blocker 1, the normative awareness layer has the right *shape* (status card, timeline, reference lists) but the wrong *confidence model*. The information architecture is sound — showing revocation signals, alteration signals, and connected references is the correct decomposition. The implementation is unsound because it doesn't distinguish between verified and inferred status.

**What's missing for investigation-grade normative awareness:**
1. **Point-in-time document reconstruction.** "What was Article 7 on March 15, 2022?" is a fundamental audit question. Without temporal versioning (valid_from/valid_to ranges on text segments, date picker in reader), historical compliance analysis is impossible. This is a backend-dependent feature but the frontend must be designed to support it.
2. **Bidirectional citation navigation.** "Cites" and "Cited by" with traversable links. The current normative_refs and procedure_refs are unidirectional (outbound only). Legal reasoning requires knowing what *later* acts reference or modify the current document.
3. **Data provenance on every metadata element.** Source URL, scrape timestamp, content checksum, last verification date. This must be visible (not hidden behind settings) in an expandable "Audit Trail" section on every document. Professional users must be able to demonstrate to courts and regulators that their data source is reliable and current.

## 4. Overall Trajectory Assessment

The product is **moving in the right direction** but is not yet on the investigation-grade track. Here is the honest positioning:

| Capability Layer | Current State | Investigation-Grade Threshold |
|---|---|---|
| Navigation / Shell | ✅ Solid | Adequate |
| Document Reading | ✅ Strong (TOC, progress, deep-link, export) | Adequate with minor gaps |
| Normative Awareness | ⚠️ Present but untrustworthy | Requires confidence model + provenance |
| Search Structure | ⚠️ Basic facets, no citation parsing | Requires citation parser + Boolean + saved searches |
| Query Transparency | ⚠️ Good data model, bad UX (auto-apply) | Requires user confirmation of inferred filters |
| Temporal Analysis | ❌ Absent | Requires point-in-time reconstruction |
| Workspace / Case Management | ❌ Absent | Requires saved searches + folders + alerts |
| Data Provenance / Audit Trail | ❌ Absent | Requires visible lineage on all metadata |

The iteration improved the **floor** (consistent navigation, basic transparency) without raising the **ceiling** (trustworthy legal data, professional workflow support). The product currently serves a "legal-curious generalist" — someone who wants to browse DOU documents with some structure. It does not yet serve the "investigating auditor" or "compliance attorney" who needs to build a defensible research record.

## 5. Recommended Next Steps, Prioritized by Impact

### Priority 1 — Fix the Blockers (Sprint-level, 1–2 weeks)
1. **Normative status confidence model**: Implement the three-tier display (Verified / Unverified / Unknown) with provenance footer. This is primarily a frontend refactor of the existing ValidityCard component plus a backend contract for explicit metadata quality signals.
2. **Inferred filters as suggestions, not constraints**: Refactor FilterPanel to present inferred filters as dismissible suggestions. Add query audit log export.
3. **Mobile document route navigation**: Implement context bar or persistent nav. Preserve search state across navigation.

### Priority 2 — Citation Parsing (2–3 weeks, highest single-feature ROI)
Implement a citation parser in the search bar supporting standard Brazilian legal citation formats (Lei, Decreto, Portaria, IN, RDC, etc.). Auto-populate structured filters from parsed citation. On exact match, navigate directly to document with article-level anchor. This single feature changes the product's identity from "search engine for DOU" to "legal research tool."

### Priority 3 — Data Provenance Layer (1–2 weeks)
Add an expandable "Trilha de auditoria" section to every document page showing: source URL, scrape/sync timestamp, content hash, last verification date. Expose in API types and render in reader. Low effort, high trust signal for professional users.

### Priority 4 — Temporal Versioning (Backend-dependent, 4–8 weeks)
Design the frontend for point-in-time document reconstruction: date picker in reader header, version diff view, temporal query parameter in search. This requires significant backend investment (temporal text storage, version indexing) but the frontend should be designed now to accommodate it. This is the feature that transitions the product from document browser to legal research platform.

### Priority 5 — Workspace / Case Management (4–6 weeks)
Saved searches, case folders, annotation/bookmarking within documents, search alert subscriptions. This requires authentication infrastructure. Defer implementation but begin data model design now — localStorage-based recent history is a stepping stone, not a destination.

---

## Final Verdict

**Not yet production-ready for professional legal/audit users.** The three implemented changes are structurally sound but carry two trust-destroying flaws (heuristic validity presented as status, inferred filters auto-applied) that must be corrected before any professional user exposure. After those fixes, citation parsing is the single feature that most changes the product's competitive position. The architecture is heading in the right direction; the confidence model and workflow support need to catch up to the information architecture ambitions.
