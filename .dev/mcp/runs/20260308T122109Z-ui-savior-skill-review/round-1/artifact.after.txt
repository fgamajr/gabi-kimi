## Accessibility Baseline (Non-Negotiable)

All recommendations must meet WCAG 2.1 AA:
- **Color contrast**: 4.5:1 for body text, 3:1 for large text and UI components
- **Focus indicators**: `focus-visible:ring-2 focus-visible:ring-offset-2` on all interactive elements
- **Keyboard navigation**: All interactive elements reachable via Tab, no keyboard traps, focus trap only in modals
- **Semantic HTML**: `<button>` for actions, `<a>` for navigation, landmark regions (`<main>`, `<nav>`, `<header>`, `<footer>`)
- **ARIA**: `aria-label` on icon-only buttons, `aria-expanded` on toggles, `aria-live` regions for dynamic content
- **Skip navigation**: Include skip-to-content link for all navigation patterns
- **Reduced motion**: Always respect `prefers-reduced-motion` — this is a design principle, not an afterthought

### Step 4 — Design Analysis

Provide a structured recommendation covering the relevant sections below. Skip sections that don't apply to the task.

#### Layout & Composition
- **Grid system**: 12-col for marketing, asymmetric for editorial, bento for dashboards/features
- **Whitespace strategy**: section padding 96–128px desktop, 48–64px mobile. Component internal padding follows 4px base scale
- **Visual hierarchy**: size contrast (2:1 minimum between heading levels), weight contrast (bold headings, regular body), color contrast (primary for actions, muted for secondary)
- **Section rhythm**: alternate content density — dense section → breathing section → dense section

#### Color & Dark Mode
- Define semantic color tokens in `tailwind.config`: `--color-surface`, `--color-text-primary`, `--color-text-muted`, `--color-border`
- Use `dark:` variants for every background, text, and border color — never hardcode `slate-50`/`slate-900` pairs
- Respect `prefers-color-scheme` for initial state; provide manual toggle
- Test contrast ratios in both modes

#### Navigation (navbar.gallery patterns)

Decision tree:
- Marketing site → **sticky, transparent-on-scroll** with CTA button visible at all breakpoints
- SaaS app → **fixed sidebar** on desktop, **bottom tab bar** on mobile
- Content/blog → **minimal top bar** with search prominent
- E-commerce → **mega menu** with category hierarchy

Mobile behavior: drawer (standard), bottom sheet (mobile-app feel), or hamburger overlay (full-screen). Collapse at `lg:1024` breakpoint.

#### Hero Sections (landing.love patterns)

Decision tree:
- Has product screenshot/demo → **split layout (text + visual)**
- Abstract/developer product → **centered with illustration or code snippet**
- Lifestyle/consumer brand → **full-bleed media**
- No visual assets yet → **gradient mesh or animated geometric background**

Structure:
- Headline: benefit-first, max 8 words
- Subheadline: supporting value prop, max 20 words
- CTA placement: primary (filled) + secondary (ghost/outline)
- Social proof strip below hero: logos, metrics, or testimonial snippet

#### Feature Sections (saaspo.com patterns)

Decision tree:
- 3–4 features → **bento grid** (asymmetric, visual weight on key feature)
- 5+ features → **icon grid** (uniform, scannable)
- Complex features needing explanation → **alternating rows** (text left/right, visual opposite)
- Features with demos → **tabbed showcase** with live preview

#### CTA Sections (cta.gallery patterns)

Decision tree:
- End of page → **contrast band** (dark bg on light page, or vice versa)
- Mid-content → **inline form** (email capture with one field)
- Persistent → **floating bar** (bottom of viewport, dismissible)
- High commitment → **modal trigger** with value summary

Copy formula: `[Action verb]` + `[Benefit]` + `[Timeframe/ease]`
Button hierarchy: primary (filled) + secondary (outline/ghost). Never two filled buttons side by side.

#### Cards & Lists (curated.design patterns)

Decision tree:
- Content with images → **media-top card**
- Horizontal scanning (feeds) → **horizontal card**
- Visual portfolio → **overlay card** with text on image
- Data/metrics → **minimal card** with stat emphasis

Hover states: `hover:-translate-y-1 hover:shadow-lg transition-all duration-200`. Never animate layout-triggering properties.

#### Pricing Sections (saaspo.com patterns)
- Layout: 3-card grid, middle card highlighted with `ring-2 ring-primary scale-105`
- Toggle: monthly/yearly with annual savings badge
- Features: checkmark list, gray out unavailable items on lower tiers
- CTA: one per card, primary on recommended tier, outline on others
- Mobile: stack vertically, keep recommended tier first

#### Form Components
- Labels: always visible above input (not placeholder-as-label — that's an anti-pattern)
- Validation: inline, real-time after first blur, error text below field in `text-destructive`
- Submit: right-aligned on desktop, full-width on mobile
- Multi-step: progress indicator, allow back navigation, persist state

#### Footer Patterns
- Marketing: multi-column links, newsletter signup, social icons, legal links
- App: minimal — version, status link, help/docs
- Mobile: accordion sections or single-column stack

#### Motion & Animation (appmotion.design patterns)

**Prefer CSS-only for simple cases.** Only reach for framer-motion when you need gesture handling, layout animations, or complex orchestration.

| Technique | CSS-only (Tailwind) | framer-motion |
|-----------|---------------------|---------------|
| Entrance fade-up | `animate-in fade-in slide-in-from-bottom-4` | `initial={{ opacity: 0, y: 16 }} animate={{ opacity: 1, y: 0 }}` |
| Hover scale | `hover:scale-[1.02] transition-transform` | `whileHover={{ scale: 1.02 }}` |
| Skeleton loader | `animate-pulse bg-muted rounded` | not needed |
| Page transition | View Transitions API | `AnimatePresence` + `motion.div` |
| Scroll reveal | `IntersectionObserver` + CSS class toggle | `whileInView` |

**Performance rules:**
- Only animate `transform` and `opacity` (GPU-composited properties)
- Avoid `will-change` unless profiling shows jank — it causes memory overhead; if used, add via JS before animation and remove after
- Use CSS `contain: layout paint` for animated sections to limit repaint scope
- Use `content-visibility: auto` for below-fold sections to reduce initial render cost
- Durations: micro-interactions 150ms, reveals 250–300ms, page transitions 350–400ms. Nothing over 500ms.
- `prefers-reduced-motion: reduce` → disable all motion except opacity fades

#### Responsive Strategy

- **Marketing/public pages**: mobile-first (`sm:` → `md:` → `lg:` → `xl:`)
- **Admin dashboards/data-dense interfaces**: desktop-first, then scale down with responsive overrides
- **Component-level**: use `@container` queries (Tailwind `@container` variant) for cards, sidebar widgets, and dashboard panels — superior to viewport queries when components render in varying parent widths
- Touch targets: minimum 44×44px (`min-h-11 min-w-11`)
- Typography: fluid `clamp(1rem, 2.5vw, 1.25rem)` for body, responsive prefixes for headings
- Images: `<picture>` with `srcset`, serve WebP/AVIF with JPEG fallback, always set `aspect-ratio` or explicit `width`/`height` to prevent CLS
- Data-dense components (pricing tables, comparison grids): switch to accordion or swipeable cards on mobile — never horizontal scroll
- Navigation: drawer or bottom-sheet on mobile, full nav on desktop

## Design Principles (Always Apply)

1. **Contrast creates hierarchy** — Use size, weight, and color to guide the eye. Minimum 2:1 size ratio between heading levels.
2. **Whitespace is a feature** — Generous padding signals quality; cramped layouts signal amateur. When in doubt, add more space.
3. **One primary action per view** — Every screen has ONE thing you want users to do. Everything else is secondary or tertiary.
4. **Motion with purpose** — Animate to guide attention, confirm actions, or show spatial relationships. Never animate for decoration.
5. **Design for the primary device first** — Mobile-first for marketing and public-facing pages. Desktop-first for admin dashboards and data-dense interfaces. Never bolt on responsiveness as an afterthought.
6. **Consistency over creativity** — Reuse patterns within a product. Novelty per-component is noise. Establish a system, then follow it.
7. **Performance is UX** — A fast, simple design beats a slow, beautiful one. Target LCP < 2.5s, CLS < 0.1, INP < 200ms.
8. **Accessibility is not optional** — If it can't be used with a keyboard, a screen reader, or in high-contrast mode, it's broken.

## Z-Index Scale (Use This)

Establish this in your project and never deviate:
- **Base content**: `z-0`
- **Sticky headers/nav**: `z-40`
- **Dropdowns/popovers**: `z-50`
- **Modals/dialogs**: `z-60`
- **Toast notifications**: `z-70`
- **Tooltips**: `z-80`

Define these in Tailwind config as `zIndex` theme extension.

## Anti-Patterns (Never Recommend)

- Hero carousels for critical content (engagement drops 90%+ past slide 1; static carousels for testimonials are acceptable)
- Hamburger menus on desktop (hides primary navigation — defeats its purpose)
- Autoplay video or audio (hostile UX, burns bandwidth)
- Light gray text on white backgrounds — or any contrast below WCAG AA (4.5:1 body, 3:1 large text)
- More than 2 typefaces (visual noise; 1 is often enough)
- Animations longer than 500ms (feels sluggish) or `will-change` applied permanently (memory bloat)
- Custom scrollbars (break platform expectations, hurt accessibility)
- Infinite scroll without "back to top" button and URL-persisted scroll position
- Images without explicit `width`/`height` or `aspect-ratio` (causes CLS)
- Unmanaged z-index (leads to stacking bugs — use the scale above)
- Inline React `style` objects when Tailwind utilities exist (breaks design system consistency, defeats purging)
- Placeholder text as the only label (disappears on input, fails accessibility)
- Modal stacking (modal over modal) — redesign the flow instead
- Client-side-only rendering for above-the-fold marketing content (hurts LCP — consider SSR or static pre-render)
- Ignoring dark mode entirely — always define both light and dark color tokens via `dark:` variants
- `<div>` as buttons or links (no keyboard/screen reader semantics)
- Horizontal scroll tables on mobile (use accordion, cards, or responsive reformatting)
