## Accessibility Baseline (Non-Negotiable)

All recommendations must meet WCAG 2.1 AA:
- **Color contrast**: 4.5:1 for body text, 3:1 for large text and UI components. Color must never be the **only** indicator of state — always pair with icon, text, or shape change.
- **Focus indicators**: Standardized across all code — `focus-visible:ring-2 focus-visible:ring-offset-2 focus-visible:ring-blue-600 dark:focus-visible:ring-blue-400`. On dark backgrounds, add `focus-visible:ring-offset-slate-900 dark:focus-visible:ring-offset-white` (or appropriate background token).
- **Keyboard navigation**: All interactive elements reachable via Tab, no keyboard traps, focus trap only in modals/drawers (with Escape to dismiss)
- **Semantic HTML**: `<button>` for actions (never `<div onClick>`), `<a>` for navigation, landmark regions (`<main>`, `<nav>`, `<header>`, `<footer>`)
- **Nesting**: Never nest interactive elements (no `<button>` inside `<a>`, no `<a>` inside `<button>`)
- **ARIA**: `aria-label` on icon-only buttons, `aria-expanded` on toggles, `aria-live="polite"` for dynamic content, `aria-describedby` for form errors, `role="alert"` for toast notifications
- **Skip navigation**: Include skip-to-content link for all navigation patterns. The skip link must have dark mode support.
- **Reduced motion**: Always respect `prefers-reduced-motion` — listen reactively for changes, not just on mount

### Step 2 — Gather Context

Before making design recommendations, gather repo context. Search adaptively — do not assume a specific project structure. **Keep this fast:** if a file search yields no results quickly, use defaults and move on.

1. **Tailwind configuration** — search for:
   - `tailwind.config.{js,ts,mjs,cjs}` at project root
   - CSS files containing `@theme` blocks (Tailwind v4 CSS-based config)
   - CSS files containing `@import 'tailwindcss'` (Tailwind v4 entry point)
   - `postcss.config.*` with `@tailwindcss/postcss` plugin
   - `vite.config.*` with `@tailwindcss/vite` plugin (Tailwind v4 signal)
   - Extract: custom theme tokens (colors, spacing, fonts), dark mode strategy, plugins installed
2. **Component inventory** — search for `.tsx`/`.jsx` files in: `src/components/`, `src/app/`, `app/`, `components/`, `lib/`, or any directory with React component exports. Note naming conventions and existing patterns.
3. **package.json** (project root, or nearest in monorepo — check for `workspaces` field) — identify:
   - Router: `react-router`, `@tanstack/router`, or none
   - Animation: `framer-motion`, `motion`, `react-spring`, or none
   - UI library: `@radix-ui/*` (shadcn/ui), `@headlessui/*`, or none
   - Plugins: `tailwindcss-animate`, `@tailwindcss/container-queries`
4. **shadcn/ui detection** — look for `components.json` at root (shadcn config file)
5. **Design tokens** — check for CSS custom properties in global CSS files

**Fallbacks if files are missing:**
- No Tailwind config → assume default Tailwind with no custom tokens
- No components directory → greenfield project; recommend `components/ui/` for atoms, `components/sections/` for composites
- No animation library → default to CSS-only animations
- No UI library → use vanilla Tailwind + React

**Tailwind version detection (check in order):**
1. `@import 'tailwindcss'` in CSS entry file → **Tailwind v4**
2. `@tailwindcss/vite` in vite.config → **Tailwind v4**
3. `@tailwindcss/postcss` in postcss.config → **Tailwind v4**
4. `@theme` in CSS → **Tailwind v4** (CSS-based config, `@variant dark`, `@utility` directives)
5. `tailwind.config.*` with JS/TS → **Tailwind v3** (JS config, `darkMode: 'class'`, plugin system)
6. None found → assume **Tailwind v3** with defaults

**Monorepo detection:** If `workspaces` in package.json, `packages/` or `apps/` directories exist, identify which package the user is working in. Components should live within that package's source, not at monorepo root.

This context determines all code examples and recommendations below.

### Step 4 — Design Analysis

Provide a structured recommendation covering only sections relevant to the task.

#### Layout & Composition
- **Grid system**: 12-col for marketing, asymmetric for editorial, bento for dashboards/features
- **Whitespace strategy**: section padding 96–128px desktop, 48–64px mobile. Component internal padding follows 4px base scale
- **Visual hierarchy**: size contrast (2:1 minimum between heading levels), weight contrast (bold headings, regular body), color contrast (primary for actions, muted for secondary)
- **Section rhythm**: alternate content density — dense section → breathing section → dense section

#### Color & Dark Mode
- **Tailwind v3**: Define semantic color tokens in `tailwind.config.js` `theme.extend.colors` referencing CSS custom properties
- **Tailwind v4**: Define in `@theme` block in your main CSS file
- Semantic token names: `--color-background`, `--color-foreground`, `--color-muted`, `--color-border`, `--color-primary`, `--color-primary-foreground`
- Use `dark:` variants for every background, text, and border color
- Respect `prefers-color-scheme` for initial state; provide manual toggle
- Test contrast ratios in both modes

#### Navigation

Decision tree:
- Marketing site → **sticky, transparent-on-scroll** with CTA button visible at all breakpoints
- SaaS app → **fixed sidebar** on desktop, **bottom tab bar** on mobile
- Content/blog → **minimal top bar** with search prominent
- E-commerce → **mega menu** with category hierarchy

Mobile behavior: drawer (standard), bottom sheet (mobile-app feel), or hamburger overlay (full-screen). Collapse at `lg:` (1024px) breakpoint.

**Mobile nav accessibility requirements:**
- Focus trap inside open drawer/overlay (trap Tab cycle within nav, release on close)
- Body scroll lock: `overflow: hidden` on `<html>` when nav is open
- Escape key dismissal: close nav on Escape keypress
- Trigger button: `aria-expanded="true|false"` + `aria-controls="mobile-nav-id"`
- Return focus to trigger button on close

#### Hero Sections

Decision tree:
- Has product screenshot/demo → **split layout (text + visual)**
- Abstract/developer product → **centered with illustration or code snippet**
- Lifestyle/consumer brand → **full-bleed media**
- No visual assets yet → **gradient mesh or animated geometric background**

Structure: benefit-first headline (max 8 words), supporting subheadline (max 20 words), primary CTA (filled) + secondary (ghost/outline), social proof strip below.

**Image placeholder strategy** (greenfield projects without assets):
```tsx
{/* Placeholder — replace with actual product screenshot */}
<div className="aspect-video w-full rounded-2xl bg-gradient-to-br from-slate-100 to-slate-200 dark:from-slate-800 dark:to-slate-700" />
```

#### Feature Sections

Decision tree:
- 3–4 features → **bento grid** (asymmetric, visual weight on key feature)
- 5+ features → **icon grid** (uniform, scannable)
- Complex features → **alternating rows** (text left/right, visual opposite)
- Features with demos → **tabbed showcase** with live preview

#### CTA Sections

Decision tree:
- End of page → **contrast band** (dark bg on light page, or vice versa)
- Mid-content → **inline form** (email capture with one field)
- Persistent → **floating bar** (bottom of viewport, dismissible)
- High commitment → **modal trigger** with value summary

Copy formula: `[Action verb]` + `[Benefit]` + `[Timeframe/ease]`. Never place two filled buttons side by side — primary (filled) + secondary (outline/ghost).

#### Cards & Lists

Decision tree:
- Content with images → **media-top card**
- Horizontal scanning → **horizontal card**
- Visual portfolio → **overlay card** with text on image
- Data/metrics → **minimal card** with stat emphasis

Hover: `hover:-translate-y-1 hover:shadow-lg transition-all duration-200`. Never animate layout-triggering properties (width, height, top, left).

#### Pricing Sections
- **Standard (2–3 tiers)**: 3-card grid, middle card highlighted with `ring-2 ring-primary scale-105`
- Toggle monthly/yearly with annual savings badge
- Checkmark feature list, gray out unavailable items on lower tiers
- Primary CTA on recommended tier, outline on others
- **Enterprise tier**: "Contact Sales" CTA with no price shown, positioned last
- **4+ tiers**: Use horizontal comparison table on desktop, accordion per tier on mobile
- **FAQ accordion** below pricing to handle common objections
- Mobile: stack vertically, recommended tier first

#### States & Feedback

Decision tree:
- **Empty state** (first-use): illustration + headline + single action CTA. Never show a blank screen.
- **Empty state** (no results): search suggestions + clear filters action
- **Error boundary**: friendly message + "Try again" button + optional "Report issue" link
- **Loading**: skeleton placeholders matching content shape (`animate-pulse`), never a full-page spinner
- **Toast/notification**: bottom-right on desktop, top-center on mobile, auto-dismiss 5s, `role="alert"` + `aria-live="polite"`

#### Form Components
- Labels: always visible above input (placeholder-as-label is an anti-pattern — it disappears on input, fails accessibility)
- Validation: inline, real-time after first blur, error text below field with `aria-describedby` linking error to input
- Submit: right-aligned on desktop, full-width on mobile
- Multi-step: progress indicator, allow back navigation, persist state

#### Motion & Animation

**Prefer CSS-only for simple cases.** Only reach for framer-motion when you need gesture handling, layout animations, or complex orchestration.

| Technique | Vanilla Tailwind/CSS | framer-motion |
|-----------|---------------------|---------------|
| Entrance fade-up | Custom keyframe (see below) | `initial={{ opacity: 0, y: 16 }} animate={{ opacity: 1, y: 0 }}` |
| Hover scale | `hover:scale-[1.02] transition-transform` | `whileHover={{ scale: 1.02 }}` |
| Skeleton loader | `animate-pulse bg-slate-200 dark:bg-slate-700 rounded` | not needed |
| Page transition | View Transitions API (progressive enhancement — Chrome only, safe to use as enhancement) | `AnimatePresence` + `motion.div` |
| Scroll reveal | `IntersectionObserver` + CSS class toggle | `whileInView` |

**Custom keyframe for fade-up (add to Tailwind config or CSS):**

Tailwind v3 — `tailwind.config.js`:
```js
theme: { extend: {
  keyframes: { fadeSlideUp: { '0%': { opacity: '0', transform: 'translateY(16px)' }, '100%': { opacity: '1', transform: 'translateY(0)' } } },
  animation: { 'fade-slide-up': 'fadeSlideUp 0.3s ease-out forwards' }
} }
```

Tailwind v4 — main CSS file:
```css
@theme { --animate-fade-slide-up: fadeSlideUp 0.3s ease-out forwards; }
@keyframes fadeSlideUp { from { opacity: 0; transform: translateY(16px); } to { opacity: 1; transform: translateY(0); } }
```

If `tailwindcss-animate` plugin is detected in the project, use its classes (`animate-in fade-in slide-in-from-bottom-4`) instead.

**Performance rules:**
- Only animate `transform` and `opacity` (GPU-composited properties)
- Avoid `will-change` unless profiling shows jank — add via JS before animation, remove after. `forwards` fill mode is correct for one-shot entrance animations; for repeated animations, prefer default fill mode to avoid retaining composited layers.
- Use CSS `contain: layout paint` for animated sections to limit repaint scope
- Durations: micro-interactions 150ms, reveals 250–300ms, page transitions 350–400ms. Nothing over 500ms.
- `prefers-reduced-motion: reduce` → disable all motion except opacity fades. Listen reactively with `matchMedia.addEventListener('change', ...)` — do not read once on mount.

#### Responsive Strategy

- **All Tailwind implementations are mobile-first** — unprefixed utilities are the mobile base, `sm:`/`md:`/`lg:`/`xl:` add breakpoint overrides
- For admin dashboards, **design** desktop-first (start your thinking at desktop) but **implement** mobile-first in Tailwind. Use `max-md:` variants only for specific mobile overrides where it's cleaner.
- **Container queries**: use `@container` (requires `@tailwindcss/container-queries` plugin in v3, native in v4) for cards and dashboard panels. Parent needs `@container` class for `container-type: inline-size`.
- Touch targets: minimum 44×44px (`min-h-11 min-w-11`)
- Typography: define fluid sizes in Tailwind config — e.g., `'fluid-base': ['clamp(1rem, 2.5vw, 1.25rem)', { lineHeight: '1.5' }]`
- Images: use Vite `import` for processed assets or `public/` for static serving. Always set `aspect-ratio` or explicit `width`/`height` to prevent CLS. Hero images: `fetchpriority="high"` + `loading="eager"`. Below-fold: `loading="lazy"`.
- Data-dense components: switch to accordion or swipeable cards on mobile — never horizontal scroll tables

Provide concrete Tailwind + React code for the recommended patterns.

**Color strategy based on Step 2 detection:**
- **shadcn/ui detected** (`components.json` or `@radix-ui/*`): use semantic tokens — `bg-background`, `text-foreground`, `text-muted-foreground`, `border-border`
- **Vanilla Tailwind** (no shadcn): use standard palette — `bg-white dark:bg-slate-950`, `text-slate-900 dark:text-slate-50`, `text-slate-500 dark:text-slate-400`, `border-slate-200 dark:border-slate-800`

**Focus ring standard** (all code examples use this):
- On light backgrounds: `focus-visible:ring-2 focus-visible:ring-offset-2 focus-visible:ring-blue-600`
- On dark backgrounds: add `focus-visible:ring-offset-slate-900 dark:focus-visible:ring-offset-white`
- With shadcn/ui: `focus-visible:ring-2 focus-visible:ring-offset-2 focus-visible:ring-ring`

**Animation strategy based on Step 2 detection:**
- `framer-motion` installed → use motion components
- `tailwindcss-animate` installed → use `animate-in` classes
- Neither → use custom keyframes or CSS transitions

```
  ui/          ← atoms: Button, Input, Badge, Card
  sections/    ← composites: Hero, Nav, FeatureGrid, CTA
  layouts/     ← page shells: MarketingLayout, AppLayout
lib/
  utils.ts     ← cn() helper if using class merging
```

**Reference — Sticky Nav with transparent-on-scroll (vanilla Tailwind):**

```tsx
import { useState, useEffect, useRef } from "react";

export function StickyNav() {
  const [scrolled, setScrolled] = useState(false);
  const scrolledRef = useRef(false);

  useEffect(() => {
    const onScroll = () => {
      const isScrolled = window.scrollY > 20;
      if (isScrolled !== scrolledRef.current) {
        scrolledRef.current = isScrolled;
        setScrolled(isScrolled);
      }
    };
    window.addEventListener("scroll", onScroll, { passive: true });
    return () => window.removeEventListener("scroll", onScroll);
  }, []);

  return (
    <>
      {/* Skip link — requires <main id="main"> in your page layout */}
      <a
        href="#main"
        className="sr-only focus:not-sr-only focus:absolute focus:z-50 focus:bg-white focus:text-slate-900 dark:focus:bg-slate-900 dark:focus:text-white focus:px-4 focus:py-2"
      >
        Skip to content
      </a>
      <header
        className={`fixed inset-x-0 top-0 z-40 transition-all duration-300 ${
          scrolled
            ? "bg-white/90 shadow-sm backdrop-blur-md dark:bg-slate-950/90"
            : "bg-transparent"
        }`}
      >
        <nav className="container mx-auto flex items-center justify-between px-4 py-4">
          <a href="/" className="text-xl font-bold text-slate-900 dark:text-white">
            {/* TODO: Replace with your brand name or logo */}
            Brand
          </a>
          <div className="hidden items-center gap-6 md:flex">
            <a href="#features" className="text-sm text-slate-600 hover:text-slate-900 dark:text-slate-300 dark:hover:text-white">
              Features
            </a>
            <a href="#pricing" className="text-sm text-slate-600 hover:text-slate-900 dark:text-slate-300 dark:hover:text-white">
              Pricing
            </a>
            <button className="rounded-lg bg-slate-900 px-4 py-2 text-sm font-medium text-white hover:bg-slate-800 focus-visible:ring-2 focus-visible:ring-offset-2 focus-visible:ring-blue-600 dark:bg-white dark:text-slate-900 dark:hover:bg-slate-100">
              Get Started
            </button>
          </div>
        </nav>
      </header>
    </>
  );
}
```
