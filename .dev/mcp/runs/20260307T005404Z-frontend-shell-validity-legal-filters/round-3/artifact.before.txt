# Frontend Audit — DOU Research Tool (Iteration Review)

## Scope

Three implemented changes evaluated: (1) shared app shell with route-level layout, (2) validity/version awareness in the document reader, (3) structured legal search filters. Assessed against the standard of an investigation-grade interface for professional legal and audit workflows.

---

## 1. What Actually Improved

### App Shell + Route-Level Layout
**Real gain.** A shared shell with persistent navigation is structural prerequisite infrastructure. Desktop left rail and mobile bottom nav give researchers stable orientation across routes. This is a genuine architecture improvement, not cosmetic — it means every future feature has a consistent mount point and the product stops feeling like disconnected pages.

### Structured Search Filters
**Partial gain.** Exposing section, act type, issuing organ, and date range as direct fields is a meaningful step above free-text-only search. Surfacing backend query interpretation (interpreted_query, inferred_filters, applied_filters) is a strong transparency move that most legal tools lack. Filter chips with issuing organ make active constraints visible. However, the filter set is incomplete for professional use (see blockers below).

### Normative Timeline + Validity Card
**Conceptual gain, liability risk in execution.** The idea of showing normative status and a timeline of connected references is exactly what differentiates a research tool from a document viewer. The type system expansion (normative_refs, procedure_refs) is sound data modeling. But the implementation introduces a serious professional risk that must be resolved before production.

---

## 2. Production Blockers (Ranked)

### BLOCKER 1 — Heuristic Normative Status is a Liability Hazard (Critical · Production Gate)

The "Situação normativa" card uses client-side text heuristics to infer revocation, replacement, alteration, and complementation status. This is unacceptable for professional legal use. A false-positive "Vigente" on a revoked act, or a missed revocation signal, exposes users to citing invalid law. No amount of "cautious" framing eliminates this risk when the UI renders colored badges that visually assert a definitive status.

**This blocker is a hard production gate. The normative status feature must not ship with heuristic inference. If authoritative data integration proves infeasible within the release window, the status indicator must be removed entirely — not shipped with heuristics. A blank field prompts manual verification; a green badge suppresses it.**

**Go/no-go criteria:**

1. Authoritative data integration is viable (see feasibility assessment below) → ship tri-state model.
2. Authoritative data integration is not viable → remove the status card from the release. Retain the underlying data model and heuristic signals only in a developer/debug view, never user-facing.

**Required fix (if go):** Replace the heuristic inference with a tri-state model sourced from authoritative data:

- **Vigente (Consolidado Oficial)** — only when backed by an official consolidated feed (Planalto, LexML, or equivalent backend enrichment with explicit source attribution and last-verified timestamp).
- **Revogado (Consolidado Oficial)** — same authoritative sourcing.
- **Status não verificado** — default state when no authoritative data exists. Render as a neutral alert (not colored) with explicit language: "Status normativo não verificado. Consulte o texto consolidado oficial." Link directly to the Planalto consolidated page for the act when possible.

Heuristic signals (detected revocation keywords, amendment references) may be retained only as supplementary indicators clearly labeled as "Sinais detectados no texto (não verificados)" — visually subordinate to the authoritative status, never rendered as badges, and never green/red. They are research aids, not assertions.

The normative timeline should similarly distinguish between authoritative chain links (sourced from official consolidation data) and inferred connections (parsed from document text), with clear visual differentiation.

**Technical feasibility assessment (required before committing to tri-state implementation):**

The prescribed solution assumes availability of authoritative consolidated feeds. Before implementation, the team must evaluate:

1. **API availability:** Planalto's legislative consolidation data (via LexML/SPARQL endpoint or scraping) — what endpoints exist, what rate limits apply, whether bulk validation is feasible.
2. **Coverage statistics by act type and year:** Pre-digital acts (pre-~2000) likely have limited or no structured consolidation data. If authoritative validation is achievable for <50% of indexed acts, the "Status não verificado" state dominates the UI, which risks training users to ignore the warning — a different failure mode but still dangerous. In this scenario, the feature should not ship at all; remove it rather than provide a predominantly unverified experience.
3. **Update latency:** If official sources update with 24–48 hour delays, the system may show stale status. The UI must display a "Verificado em: [date/time]" timestamp and refresh interval to make latency visible to the user.
4. **Fallback strategy:** If the primary authoritative source is unavailable, the system must degrade to "Status não verificado" — never fall back to heuristics silently.

**Liability risk quantification:** While precise error rates for the heuristic approach require empirical measurement, the risk profile is asymmetric. The probability of heuristic error is non-trivial (revocation language varies, indirect amendments are common, temporal revocations create edge cases). The severity of a legal citation error ranges from professional embarrassment to malpractice liability, regulatory non-compliance, or incorrect judicial filings. User exposure scales with adoption. Even a low error rate (e.g., 5% of status inferences incorrect) applied across professional users making consequential decisions creates unacceptable aggregate risk. This is why the go/no-go gate is binary: authoritative data or no feature.

### BLOCKER 2 — Search Filters Are Too Shallow for Professional DOU Research (High)

The current filter set covers the basics but misses metadata fields that professional researchers depend on daily:

- **DOU section parts (Seção 1, 2, 3, Extra)** — not just "section" generically. Researchers filter by these constantly; regulatory acts concentrate in Seção 1, personnel acts in Seção 2, judicial in Seção 3.
- **Publication type (original vs. retificação/republicação)** — critical for audit workflows where the distinction between original publication and correction changes the legally operative text.
- **Classification codes** — used by specialized researchers tracking specific regulatory domains.

**On boolean/proximity operators:** The audit initially prescribed boolean operators (AND/OR/NOT, NEAR/ADJ) as "table stakes." This warrants nuance. The product already invests in interpreted natural language queries (Blocker 3), which is the direction modern legal search tools (Casetext, Google Scholar) are trending. Forcing boolean syntax as a primary interface may degrade usability for researchers who benefit from the interpreted model. However, professional investigation workflows — particularly exhaustive searches where a researcher must certify completeness — still require deterministic query control that natural language interpretation cannot guarantee.

**Required fix:** Expand the filter panel in two tiers:
- **Primary panel:** Current fields plus DOU section part (Seção 1/2/3/Extra) and publication type (original/retificação/republicação). These are metadata filters, not query syntax — they belong in the primary interface for all user personas.
- **Advanced toggle:** Classification codes, "search within results" for iterative refinement, and an optional boolean syntax mode for power users who need deterministic query control. Frame this explicitly as "Busca avançada — controle manual de operadores" to signal that it overrides the interpreted model. Do not force boolean syntax on users who prefer natural language; let the two models coexist with clear framing.

Validate with user research whether the interpreted query model is meeting precision needs for the core user base before prioritizing proximity operators (NEAR/ADJ) — these may be unnecessary if semantic search quality is high.

### BLOCKER 3 — Query Interpretation Is Read-Only (High)

Surfacing interpreted_query and inferred_filters is good. Making them non-interactive is a trap. When a researcher sees the system misinterpreted "STF" as something other than the Supremo Tribunal Federal, or inferred an incorrect date range, they currently have no recourse except rewriting the entire query. This transforms a transparency feature into a frustration feature.

**Required fix:** Make inferred filter chips dismissible (click X to remove) and editable (click to modify the value). Add a "Reset interpretation" action that clears all inferred filters and re-runs the raw query. This is not a large UI effort — the chips already exist — but the interaction model must be bidirectional.

### BLOCKER 4 — localStorage as Primary Persistence (High)

Reading history, saved positions, and research context stored exclusively in localStorage is fundamentally incompatible with professional workflows:

- Device-bound: researchers move between office desktop, laptop, and mobile.
- Ephemeral: cleared by browser maintenance, private browsing, or IT policies.
- No audit trail: compliance and litigation work requires provenance — who searched what, when, with which filters, which documents were reviewed.
- No collaboration: team-based research is invisible.

**Required fix (phased):**

- **Phase 1 (immediate):** Add a clear product-level disclaimer that local storage is not persistent. Do not present localStorage-backed features (recent history, saved position) as reliable workspace capabilities. Label them explicitly as "Local device only — not synced."
- **Phase 1.5 (bridge):** Implement a hybrid persistence strategy that does not require full authentication infrastructure. Options include: shareable/bookmarkable URLs that encode search state (query, filters, sort, pagination) so that search sessions are portable via URL; optional export/import of local research state as encrypted JSON files; or an anonymous sync token (similar to Firefox Sync pairing codes) that enables cross-device state transfer without accounts. These are lower-lift than full authentication and address device portability for solo researchers.
- **Phase 2 (next cycle):** Implement authenticated sessions with server-side persistence. Model research around Project/Workspace entities. Persist search sessions (query, filters, results reviewed, timestamps) and reading state per-project. Provide exportable audit logs (search methodology receipts) for case files.

localStorage becomes the unauthenticated fallback, not the primary strategy.

### BLOCKER 5 — Mobile Document Reading Breaks Iterative Workflow (Medium)

Hiding mobile bottom navigation during document reading prioritizes an immersive reading metaphor. This is a defensible pattern for long-form content consumption (Kindle, Acrobat Reader, etc.), and the audit's initial recommendation to force persistent bottom nav was overcorrective — persistent chrome reduces viewport area and creates tap-target conflicts during dense legal text review.

However, the underlying workflow concern is real: legal research is iterative (search → open → cross-reference → return to results → open another), and browser-back navigation loses filter state, scroll position, and cognitive context.

**Required fix:** Preserve the immersive reading mode but solve the workflow problem:

1. **Guaranteed state recovery:** Persist search state (query, filters, scroll position, pagination) in sessionStorage so that browser-back and gesture-based navigation restore results exactly as left. This is the highest-priority fix — if state recovery works, the nav problem is 80% solved.
2. **Quick-return affordance:** Implement a lightweight, non-intrusive "Return to results" floating button or edge-swipe gesture rather than full persistent navigation. This preserves reading focus while providing a one-tap escape hatch.
3. **Context breadcrumb:** Show a minimal top bar on document routes indicating the originating search context (e.g., "3 of 47 results for 'ANVISA resolução 2024'") with left/right navigation to adjacent results, enabling serial document review without returning to the results list.

## 3. Depth Assessment

### Search Panel: Not Yet Investigation-Grade

The current implementation handles the "casual lookup" use case well: a researcher with a rough idea of what they want can find it faster than the raw DOU portal. Investigation-grade search requires precision — exhaustive metadata filters, reproducible search sessions, and iterative refinement. The gap is the difference between "I found something" and "I can certify I found everything relevant." The tool is currently at the former; professional workflows demand the latter. The natural language interpretation approach is promising and aligns with modern legal search trends, but it must be complemented by deterministic query controls for exhaustive search scenarios.

### Normative Awareness Layer: Architecturally Sound, Executionally Dangerous

The data model (normative_refs, procedure_refs, timeline) is the right foundation. The type system captures the right entities. But rendering heuristic inferences as status indicators — without explicit sourcing, confidence levels, or authoritative data — creates a product that is more dangerous than one with no status indicators at all. A blank field prompts the researcher to check manually. A green "Vigente" badge suppresses that instinct. The architecture should be preserved; the presentation layer must either be rebuilt around the tri-state authoritative model or removed from the release.

## 4. Accessibility Compliance Gap

The audit would be incomplete without evaluating accessibility, which is mandatory for government-facing legal tools and a likely requirement for public sector procurement (Brazilian accessibility law, Lei 13.146/2015, and WCAG 2.1 AA).

**Identified risks in the current implementation:**

- **Color-dependent status indicators (WCAG 1.4.1):** The normative status badges (green for Vigente, red for Revogado) rely solely on color to convey meaning. If the tri-state model ships, every status must include a text label and an icon (e.g., checkmark, X mark, question mark) in addition to color. Color alone is insufficient.
- **Keyboard operability for filter panel (WCAG 2.1.1):** The structured filter panel (dropdowns, date pickers, chips) must be fully operable via keyboard without mouse dependency. Tab order, focus management on chip creation/dismissal, and date picker keyboard interaction need explicit testing.
- **Screen reader compatibility for normative timeline (WCAG 4.1.2):** The timeline visualization requires ARIA labels for each node (act type, date, relationship type) and a linearized alternative for screen readers. A visual graph without semantic markup is invisible to assistive technology.
- **Query interpretation chips (WCAG 4.1.2):** Interactive filter chips must expose name, role, and value to assistive technology. Dismissible chips need aria-label describing the filter being removed and appropriate role (e.g., `role="option"` with `aria-selected`, or a button with descriptive label).

**Required:** Add accessibility testing (automated via axe-core in CI, manual via screen reader testing) as a gating criterion for all UI components before production release. Prioritize the status indicator and filter panel — these are the highest-interaction-frequency components.

---

## 5. Cross-Reference Navigation Gap

The types include normative_refs and procedure_refs, but the current implementation does not describe how these render as interactive elements. For professional use, every reference to another act, article, or procedure must be a navigable link. Without this, the reference data is informational but not operational — researchers still have to manually search for connected documents.

This is not a production blocker for the current release — the tool is usable without it — but it is a Phase 3 requirement for investigation-grade status. Clicking a normative reference should open the target document (ideally in a side panel or new tab to preserve context) with a breadcrumb trail showing the reference chain. This directly enables the "follow the chain" workflow that distinguishes research tools from document viewers.

**Classified as:** Phase 3 enhancement (not a release gate, but required for investigation-grade designation).

## 6. Sequenced Roadmap

### This cycle (release gate):

1. **Normative status resolution (hard gate):** Complete feasibility assessment of authoritative data sources. If viable (>50% coverage, API available, latency <24h), implement tri-state model with source attribution and verification timestamps. If not viable, remove the status card from the release entirely. No third option.
2. **Accessibility baseline:** Audit and fix color-dependent status indicators, keyboard operability for filters, and ARIA labeling for interactive chips. Add axe-core to CI pipeline.
3. **Mobile state recovery:** Implement sessionStorage-based search state preservation so browser-back restores results exactly.

### Next cycle:

4. **Expanded search filters:** DOU section parts, publication type, advanced toggle with boolean syntax and search-within-results.
5. **Interactive query interpretation:** Editable/dismissible inferred filter chips with reset action.
6. **Hybrid persistence bridge:** URL-encoded search state for shareability/portability, optional state export/import.

### Following cycle:

7. **Authenticated workspaces:** User accounts, server-persisted search sessions, project/workspace entities, exportable audit logs (search methodology receipts).
8. **Cross-reference navigation:** One-click navigation from normative/procedure refs to target documents with context preservation.
9. **Version diff viewer and citation network foundations.**

---

## 7. Verdict

The product is moving toward an investigation-grade interface, but it is not there yet and must not ship in its current state.

The app shell and structured search represent genuine architectural progress. The normative awareness layer has the right data model but the wrong presentation contract — it is a **hard production gate** that must be resolved by either integrating authoritative data or removing the feature. There is no acceptable middle ground where heuristic status assertions reach professional users. The liability risk is asymmetric: a low probability of heuristic error multiplied by high severity of legal citation consequences across a growing user base creates unacceptable aggregate exposure.

The localStorage dependency caps the product at "personal convenience tool." The search filters need one more tier of depth. The accessibility gaps must be closed for government-facing deployment.

The direction is correct. The velocity is reasonable. The release gate is singular and non-negotiable: resolve the normative status card before production.
