1. FUNDAÇÃO: TEMA, TOKENS & STATE MANAGEMENT (dark-first)

1.1 Estratégia de tema
- tailwind.config: darkMode: 'class'
- Provider de tema: hook customizado com localStorage + matchMedia(prefers-color-scheme). Sem dependência de next-themes (app é Vite puro). Classe .dark no <html> como padrão inicial (dark-first).
- Flash prevention: script bloqueante inline no <head> que lê localStorage antes do primeiro paint, ANTES de qualquer <link> de CSS:

  <script>
    (function(){
      var t = localStorage.getItem('gabi-theme');
      var d = t === 'dark' || (!t && matchMedia('(prefers-color-scheme:dark)').matches);
      document.documentElement.classList.toggle('dark', d);
    })();
  </script>

1.2 Paleta customizada (globals.css, formato HSL)

Formato: valores HSL sem função envolvente, compatíveis com shadcn/ui (consumidos via hsl(var(--token)) no Tailwind). Definidos em blocos :root e .dark separados.

:root {
  --background:          0 0% 100%;
  --foreground:          240 10% 3.9%;
  --card:                0 0% 98%;
  --card-foreground:     240 10% 10%;
  --primary:             215 80% 48%;
  --primary-foreground:  0 0% 100%;
  --muted:               240 5% 92%;
  --muted-foreground:    240 5% 40%;
  --border:              240 5% 87%;
  --ring:                215 80% 48%;
  --surface-document:    40 20% 98%;
  --text-legal:          0 0% 20%;
  --text-legal-muted:    0 0% 50%;
  --highlight-search:    50 100% 50%;
}

.dark {
  --background:          240 10% 3.9%;
  --foreground:          0 0% 93%;
  --card:                240 10% 6%;
  --card-foreground:     0 0% 90%;
  --primary:             215 80% 58%;
  --primary-foreground:  0 0% 100%;
  --muted:               240 5% 15%;
  --muted-foreground:    240 5% 55%;
  --border:              240 5% 18%;
  --ring:                215 80% 58%;
  --surface-document:    240 8% 5%;
  --text-legal:          0 0% 82%;
  --text-legal-muted:    0 0% 55%;
  --highlight-search:    50 100% 50%;
}

Uso de --highlight-search: sempre via Tailwind com opacidade variável — bg-highlight-search/15 para resultado normal, bg-highlight-search/30 para hover. NUNCA embutir alpha no token.

Regra: NÃO usar branco puro (#fff / text-white) para texto em dark mode — máximo zinc-200 para body, zinc-300 para texto legal longo. Isso reduz fadiga visual.

1.3 Contraste
- Todos os pares foreground/background devem passar WCAG 2.1 AA (4.5:1 texto normal, 3:1 texto grande).
- Validar com ferramenta automatizada (axe-core) em CI para ambos os temas.

1.4 State Management (decisão arquitetural obrigatória)

- Server state (documentos, busca, publicações): TanStack Query (React Query). Cache, dedup, retry, stale-while-revalidate — tudo embutido. NENHUM fetch manual com useEffect.
- Client state global (tema, preferências de fonte, sidebar aberta/colapsada): Zustand. Store única src/stores/app-store.ts, persisted via zustand/middleware persist para localStorage.
- Client state local (form inputs, UI transiente): useState/useReducer no componente. Não precisa de store global.
- Regra: se o estado precisa sobreviver a navegação entre rotas → Zustand. Se vem do servidor → TanStack Query. Se é efêmero → local state. Zero ambiguidade.

1.5 Error Boundaries

- ErrorBoundary (via react-error-boundary) wrapping CADA rota no router.
- Fallback UI: componente <RouteErrorFallback> com Alert do shadcn, botão retry, e log automático para Sentry.
- ErrorBoundary adicional no Document Viewer envolvendo o container de seções virtualizadas (crash de virtualização não derruba TOC nem shell).
- Implementação: src/components/error/RouteErrorFallback.tsx e src/components/error/SectionErrorFallback.tsx.

2. TIPOGRAFIA & DENSIDADE PARA DOCUMENTOS LONGOS

2.1 Font stack & estratégia de carregamento

Fontes:
- UI (menus, labels, botões): Inter Variable, system-ui, sans-serif
- Corpo de documento legal: Opcional serif — Literata Variable, Georgia, serif. Configurável via toggle do usuário (armazenado via Zustand persist).

Carregamento (crítico para LCP e CLS):
- Inter: <link rel="preload" href="/fonts/inter-variable.woff2" as="font" type="font/woff2" crossorigin> no <head>, antes do CSS.
- @font-face com font-display: swap E size-adjust/ascent-override para system-ui fallback matching:

  @font-face {
    font-family: 'Inter Variable';
    src: url('/fonts/inter-variable.woff2') format('woff2');
    font-display: swap;
  }
  @font-face {
    font-family: 'Inter Fallback';
    src: local('system-ui');
    ascent-override: 90%;
    descent-override: 22%;
    line-gap-override: 0%;
    size-adjust: 107%;
  }

- Literata: NÃO preload. Carregada via lazy @font-face injetado somente quando usuário ativa toggle serif (import dinâmico do CSS). Fallback: Georgia imediato.

3. MOTION BUDGET (React Bits)

3.1 Regras inegociáveis
- Máximo 2 animações simultâneas no viewport.
- Micro-interações (hover, press, toggle): ≤200ms.
- Transições de entrada (cards, seções): ≤300ms.
- Transições de página/sheet: ≤500ms.
- Easing padrão: cubic-bezier(0.16, 1, 0.3, 1) para entradas; cubic-bezier(0.4, 0, 0.2, 1) para saídas.
- prefers-reduced-motion: media query respeitada. Hook useReducedMotion() wrapa todo uso de React Bits — se ativo, render estático sem animação. Zero exceções.

Implementação mínima do hook:

  function useReducedMotion(): boolean {
    const [reduced, setReduced] = useState(() =>
      window.matchMedia('(prefers-reduced-motion: reduce)').matches
    );
    useEffect(() => {
      const mq = window.matchMedia('(prefers-reduced-motion: reduce)');
      const handler = (e: MediaQueryListEvent) => setReduced(e.matches);
      mq.addEventListener('change', handler);
      return () => mq.removeEventListener('change', handler);
    }, []);
    return reduced;
  }

3.2 Isolamento de React Bits
- NUNCA importar React Bits diretamente em componentes de tela.
- Criar wrappers em src/components/motion/: AnimatedCard, RevealSection, StaggerList.
- Cada wrapper aceita prop animate={boolean} (default: true, override por useReducedMotion).
- Se React Bits for descontinuado, substituímos apenas esses wrappers.
- Snapshot tests nos wrappers em CI para detectar breaking changes do React Bits antes de atingir telas.

3.3 Onde usar (e onde NÃO usar)
- SIM: entrada de cards no dashboard (stagger), reveal de seções no Document Viewer (fade-up ao scroll), transição de abertura de Sheet/Bottom Sheet.
- NÃO: dentro de listas virtualizadas, em tabelas de dados, em qualquer elemento que re-renderiza frequentemente, abaixo da fold em documentos longos (custo de IntersectionObserver em 500+ nós).

4.3 SEARCH (Busca)
──────────────────────────────────
Layout:
  Mobile: Input fixo no topo + resultados em lista vertical full-width.
  Tablet/Desktop: Input centralizado (max-width 640px) + resultados abaixo em lista, filtros em Popover lateral.

Separação de responsabilidades:
  - CommandDialog global (⌘K) = navegação rápida e atalhos. Ao digitar query de busca no CommandDialog, opção "Buscar '[query]' no DOU →" navega para /search?q=[query]. O CommandDialog NÃO é a interface de busca.
  - Tela /search = Input dedicado (componente próprio, NÃO Command) + resultados inline + filtros. É a experiência de busca completa.

Componentes shadcn/ui:
  - Input (campo de busca dedicado, persistente na página, com ícone de busca e clear button)
  - Badge (filtros ativos: tipo de ato, órgão, data)
  - Popover + Calendar (filtro de data)
  - Select (filtro de tipo de ato, órgão)
  - ScrollArea (lista de resultados)
  - Card (cada resultado: título, trecho com highlight via bg-highlight-search/15, metadados)
  - Separator (entre resultados)
  - Button (limpar filtros, carregar mais)
  - Skeleton (resultados carregando)

Data fetching: TanStack Query — queryKey ['search', { query, filters }], enabled: query.length >= 2, keepPreviousData: true para UX sem flash entre queries.

RB:
  - Nenhum nos resultados (performance > estética em lista que pode ter centenas de itens).
  - AnimatedCard apenas no primeiro carregamento dos filtros (fade-in, 200ms).

Virtualização: @tanstack/react-virtual para lista de resultados quando estimatedTotalHeight > 3x viewportHeight. Scroll infinito com IntersectionObserver no último item.

Estados: loading (Skeleton list), empty (ilustração "Nenhum resultado para [query]"), error (Alert), initial (sugestões de busca recente armazenadas em Zustand).

──────────────────────────────────
4.4 DOCUMENT VIEWER (Visualizador de Documento)
──────────────────────────────────
** Tela crítica — caso de uso primário do GABI DOU. **

Layout:
  Mobile: documento full-width, TOC acessível via Sheet (bottom sheet com snap points via Vaul: 30% / 60% / 90%).
  Tablet: documento com TOC em Sheet lateral (esquerda, 280px).
  Desktop: grid 3 colunas — TOC sticky à esquerda (240px) + documento central (max 68ch) + painel de metadados/ações à direita (200px).

Componentes shadcn/ui:
  - ScrollArea (container principal do documento — custom scrollbar estilizado)
  - Sheet + Vaul (TOC mobile e tablet, com snap points)
  - NavigationMenu (TOC desktop — sticky, anchor links)
  - Separator (entre seções/artigos do DOU)
  - Tooltip (ações inline: copiar artigo, compartilhar, favoritar)
  - Button (ações: voltar, favoritar, compartilhar, imprimir)
  - Badge (tipo do ato, órgão emissor, data de publicação)
  - Popover (menu de compartilhamento)
  - Skeleton (carregamento progressivo de seções)

Data fetching: TanStack Query — queryKey ['document', documentId], com select para transformar resposta em estrutura de seções.

TOC interativa — estratégia dual para seção ativa:

  Seções não-virtualizadas (primeiras 20, renderizadas no DOM):
  - UM ÚNICO IntersectionObserver (não um por heading) observando todos os headings renderizados, com rootMargin "-10% 0px -80% 0px" para detectar heading no topo do viewport.

  Seções virtualizadas (restantes):
  - Scroll position tracker: getScrollElement() do virtualizer + array pré-computado de offsets acumulados por seção. Binary search no array para determinar seção ativa com O(log n). Latência de atualização da TOC: < 100ms.

  Navegação TOC → seção virtualizada:
  - virtualizer.scrollToIndex(sectionIndex, { align: 'start' }) — o react-virtual renderiza a seção-alvo antes de scrollar. NÃO usar anchor links nativos para seções virtualizadas (nó não existe no DOM).
  - Anchor links nativos apenas para as primeiras 20 seções (sempre no DOM).

  Teste obrigatório: teste automatizado (Playwright) que valida highlight correto da TOC ao scrollar até seção 50+, 100+, e última seção.

Progressive rendering para documentos longos (500+ seções):
  - Paginação por seção: carregar inicialmente as primeiras 20 seções renderizadas no DOM.
  - @tanstack/react-virtual com measureElement para seções restantes (alturas variáveis). Overscan de 5 itens para scroll suave.
  - Critério de ativação: virtualizar quando itemCount * estimatedAverageHeight > viewportHeight * 3 (não threshold arbitrário de contagem).
  - Fallback: botão "Carregar mais seções" se virtualização falhar em device low-end (detectado via navigator.deviceMemory < 4 ou fallback após crash no ErrorBoundary de seções).

ErrorBoundary específico: envolve APENAS o container de seções virtualizadas. Crash de virtualização mostra <SectionErrorFallback> ("Erro ao renderizar seções. Tentando modo simplificado...") e fallback para paginação estática. TOC e shell continuam funcionais.

Busca in-doc:
  - Atalho: / (slash) para abrir busca in-doc (padrão de readers como PDF.js). NÃO interceptar Ctrl+F/⌘F — deixar busca nativa do browser funcionando como fallback. Usuários power-user esperam o comportamento nativo.
  - UI: Input flutuante no topo do documento com contagem de matches e setas prev/next.
  - Highlight: bg-highlight-search/15 nos matches, bg-highlight-search/40 no match ativo.

RB:
  - RevealSection: fade-up suave (200ms) ao scrollar para seções novas. Aplicar APENAS nas primeiras 20 seções renderizadas; seções virtualizadas entram sem animação.
  - Nenhuma animação em texto legal — legibilidade absoluta.

Classe .prose-legal aplicada no container de texto do documento (ver seção 2.4).

Estados: loading (Skeleton com layout simulado do documento), error (RouteErrorFallback fullscreen com retry), 404 (página dedicada "Documento não encontrado").

8. ROLLOUT INCREMENTAL

Feature flags — modelo híbrido:

  Build-time (VITE_FEATURE_*): usados para excluir código inteiro de features do bundle em builds de produção. Requer rebuild+redeploy para alterar. Tempo estimado de rollback via revert+CI: ~5 minutos.

  Runtime (/config/features.json): JSON estático servido via CDN, fetched no boot do app (antes do primeiro render, com cache de 60s). Para toggle instantâneo sem rebuild. Estrutura:
    { "viewer": true, "search": true, "motion": true }
  Hook: useFeatureFlag('viewer') lê desse JSON (cacheado em memória via TanStack Query, staleTime Infinity, refetch manual).
  Override de debug: query param ?ff_viewer=false para QA.

  Regra: build-time flags controlam se o código EXISTE no bundle. Runtime flags controlam se a feature está ATIVA. Ambos devem estar "on" para a feature funcionar.
