1. FUNDAÇÃO: TEMA, TOKENS, ROUTING & STATE MANAGEMENT (dark-first)

1.1 Estratégia de tema
- tailwind.config: darkMode: 'class'
- Provider de tema: hook customizado com localStorage + matchMedia(prefers-color-scheme). Sem dependência de next-themes (app é Vite puro). Classe .dark no <html> como padrão inicial (dark-first).

Flash prevention: script bloqueante inline no <head> que lê localStorage antes do primeiro paint, ANTES de qualquer <link> de CSS. Inclui nonce placeholder para CSP e try-catch para private browsing / localStorage desabilitado.

Sincronização com Zustand: a store Zustand usa chave dedicada 'gabi-app' via persist. O tema é TAMBÉM escrito em chave isolada 'gabi-theme-value' (string pura: 'dark' | 'light' | null) toda vez que o usuário altera tema. O script inline lê APENAS 'gabi-theme-value', evitando parsing de JSON da store.

  <script nonce="__CSP_NONCE__">
    (function(){
      try {
        var t = localStorage.getItem('gabi-theme-value');
        var d = t === 'dark' || (!t && matchMedia('(prefers-color-scheme:dark)').matches);
      } catch(e) {
        var d = matchMedia('(prefers-color-scheme:dark)').matches;
      }
      document.documentElement.classList.toggle('dark', d);
    })();
  </script>

Zustand side-effect (em useThemeStore):

  subscribe(
    (state) => state.theme,
    (theme) => {
      try { localStorage.setItem('gabi-theme-value', theme); } catch {}
      document.documentElement.classList.toggle('dark', theme === 'dark');
    }
  );

1.3 Contraste
- Todos os pares foreground/background devem passar WCAG 2.1 AA (4.5:1 texto normal, 3:1 texto grande).
- Validar com @axe-core/playwright em CI para ambos os temas, com assertions nos pares de tokens HSL específicos.
- Testar com simulação de daltonismo via Chrome DevTools (protanopia, deuteranopia, tritanopia) antes de cada release.

1.4 State Management (decisão arquitetural obrigatória)

- Server state (documentos, busca, publicações): TanStack Query (React Query). Cache, dedup, retry, stale-while-revalidate — tudo embutido. NENHUM fetch manual com useEffect.
- Client state global (tema, preferências de fonte, sidebar aberta/colapsada): Zustand. Store única src/stores/app-store.ts, persisted via zustand/middleware persist para localStorage (name: 'gabi-app').
- Client state local (form inputs, UI transiente): useState/useReducer no componente.
- Regra: se o estado precisa sobreviver a navegação entre rotas → Zustand. Se vem do servidor → TanStack Query. Se é efêmero → local state. Zero ambiguidade.

1.4.1 Padrão para Mutations (TanStack Query)

Toda mutation que afeta dados visíveis ao usuário (favoritar, compartilhar, marcar como lido) segue o padrão de optimistic update:

  useMutation({
    mutationFn: toggleFavorite,
    onMutate: async (docId) => {
      await queryClient.cancelQueries({ queryKey: ['publications'] });
      const previous = queryClient.getQueryData(['publications']);
      queryClient.setQueryData(['publications'], (old) =>
        optimisticallyToggleFavorite(old, docId)
      );
      return { previous };
    },
    onError: (_err, _docId, context) => {
      queryClient.setQueryData(['publications'], context.previous);
      toast.error('Falha ao atualizar. Tente novamente.');
    },
    onSettled: () => {
      queryClient.invalidateQueries({ queryKey: ['publications'] });
    },
  });

Regra: onMutate faz rollback otimista, onError restaura snapshot anterior, onSettled invalida para sincronizar com servidor. Toast de erro sempre visível.

1.5 Error Boundaries

- ErrorBoundary (via react-error-boundary) wrapping CADA rota no router (via errorElement — ver seção 1.6).
- Fallback UI: componente <RouteErrorFallback> com Alert do shadcn, botão retry, e log automático para Sentry.
- ErrorBoundary adicional no Document Viewer envolvendo o container de seções virtualizadas (crash de virtualização não derruba TOC nem shell).
- Implementação: src/components/error/RouteErrorFallback.tsx e src/components/error/SectionErrorFallback.tsx.
- Fallback UI: role="alert", aria-live="assertive" para leitores de tela.

Sentry:
- Dependência: @sentry/react (~15KB gzipped, tree-shaken).
- Inicialização: em src/main.tsx, ANTES de createRoot(). Sentry.init({ dsn, environment, release }).
- Source maps: upload via @sentry/vite-plugin no build de produção.
- Integração com ErrorBoundary: Sentry.withScope + captureException no onError callback do react-error-boundary.

1.6 Routing

Biblioteca: React Router v6.4+ com createBrowserRouter. Justificativa: ecossistema maduro, loader/action pattern disponível para futuro (não obrigatório agora), errorElement nativo por rota, ampla adoção da comunidade.

Árvore de rotas (src/router.tsx):

  const router = createBrowserRouter([
    {
      path: '/',
      element: <Shell />,        // layout raiz com Outlet
      errorElement: <RouteErrorFallback />,
      children: [
        {
          index: true,
          lazy: () => import('./pages/home'),   // export default Component
        },
        {
          path: 'search',
          lazy: () => import('./pages/search'),
        },
        {
          path: 'document/:documentId',
          lazy: () => import('./pages/document-viewer'),
        },
        {
          path: 'favorites',
          lazy: () => import('./pages/favorites'),
        },
        {
          path: 'profile',
          lazy: () => import('./pages/profile'),
        },
        {
          path: '*',
          lazy: () => import('./pages/not-found'),
        },
      ],
    },
  ]);

Cada rota usa lazy() do React Router v6.4 (não React.lazy) — carrega Component, loader e errorElement em um único dynamic import. Vite faz code-splitting automático por chunk.

ErrorBoundary por rota: cada módulo lazy pode exportar ErrorBoundary que sobrescreve o errorElement do Shell. Se não exportar, usa o RouteErrorFallback do layout raiz.

2. TIPOGRAFIA & DENSIDADE PARA DOCUMENTOS LONGOS

2.1 Font stack & estratégia de carregamento

Fontes:
- UI (menus, labels, botões): Inter Variable, system-ui, sans-serif
- Corpo de documento legal: Opcional serif — Literata Variable, Georgia, serif. Configurável via toggle do usuário (armazenado via Zustand persist).

Subsetting (obrigatório para performance):
- Inter Variable: subset para Latin + Latin Extended (cobre pt-BR completo). Usar pyftsubset ou glyphhanger em CI:
  npx glyphhanger --subset=Inter-Variable.woff2 --formats=woff2 --LATIN
- Resultado esperado: ~30-35KB (vs ~95KB full). Alternativa: @fontsource/inter com split por subset.
- Literata: mesmo subsetting, aplicado no momento do build.

Carregamento (crítico para LCP e CLS):
- Inter (subsetted): <link rel="preload" href="/fonts/inter-latin.woff2" as="font" type="font/woff2" crossorigin> no <head>, antes do CSS.
- @font-face com font-display: swap E fallback com ascent-override para métricas matching:

  @font-face {
    font-family: 'Inter Variable';
    src: url('/fonts/inter-latin.woff2') format('woff2');
    font-display: swap;
  }
  @font-face {
    font-family: 'Inter Fallback';
    src: local('Segoe UI'), local('SF Pro Text'), local('Ubuntu'),
         local('Roboto'), local('Helvetica Neue'), local('Arial');
    ascent-override: 90%;
    descent-override: 22%;

3. MOTION BUDGET (React Bits)

3.1 Regras inegociáveis
- Máximo 2 animações simultâneas no viewport.
- Micro-interações (hover, press, toggle): ≤200ms.
- Transições de entrada (cards, seções): ≤300ms.
- Transições de página/sheet: ≤500ms.
- Easing padrão: cubic-bezier(0.16, 1, 0.3, 1) para entradas; cubic-bezier(0.4, 0, 0.2, 1) para saídas.
- prefers-reduced-motion: media query respeitada. Hook useReducedMotion() wrapa todo uso de React Bits — se ativo, render estático sem animação. Zero exceções.

Implementação do hook (SSR-safe):

  function useReducedMotion(): boolean {
    const [reduced, setReduced] = useState(() =>
      typeof window === 'undefined'
        ? false
        : window.matchMedia('(prefers-reduced-motion: reduce)').matches
    );
    useEffect(() => {
      if (typeof window === 'undefined') return;
      const mq = window.matchMedia('(prefers-reduced-motion: reduce)');
      const handler = (e: MediaQueryListEvent) => setReduced(e.matches);
      mq.addEventListener('change', handler);
      return () => mq.removeEventListener('change', handler);
    }, []);
    return reduced;
  }

3.2 Isolamento de React Bits
- NUNCA importar React Bits diretamente em componentes de tela.
- Criar wrappers em src/components/motion/: AnimatedCard, RevealSection, StaggerList.
- Cada wrapper aceita prop animate={boolean} (default: true, override por useReducedMotion).
- Se React Bits for descontinuado, substituímos apenas esses wrappers.
- Testes de integração (NÃO snapshot) nos wrappers: validar presença/ausência de classes CSS de animação conforme prop animate. Snapshot tests são flaky com timing de animação em CI.

3.3 Onde usar (e onde NÃO usar)
- SIM: entrada de cards no dashboard (stagger), reveal de seções no Document Viewer (fade-up ao scroll), transição de abertura de Sheet/Bottom Sheet.
- NÃO: dentro de listas virtualizadas, em tabelas de dados, em qualquer elemento que re-renderiza frequentemente, abaixo da fold em documentos longos (custo de IntersectionObserver em 500+ nós).

──────────────────────────────────
** Tela crítica — caso de uso primário do GABI DOU. **

  Tablet: documento com TOC em Sheet lateral (esquerda, 280px).
  Desktop: grid 3 colunas — TOC sticky à esquerda (240px) + documento central (max 68ch) + painel de metadados/ações à direita (200px).

Componentes shadcn/ui:
  - ScrollArea (container principal do documento — custom scrollbar estilizado)
  - Sheet + Vaul (TOC mobile e tablet, com snap points)
  - NavigationMenu (TOC desktop — sticky, anchor links)
  - Separator (entre seções/artigos do DOU)
  - Tooltip (ações inline: copiar artigo, compartilhar, favoritar)
  - Button (ações: voltar, favoritar, compartilhar, imprimir)
  - Badge (tipo do ato, órgão emissor, data de publicação)
  - Popover (menu de compartilhamento)
  - Skeleton (carregamento progressivo de seções)

Data fetching: TanStack Query — queryKey ['document', documentId], com select para transformar resposta em estrutura de seções.

Prefetch inteligente: quando usuário está idle por 3s no Document Viewer, prefetch do próximo documento na lista de publicações:
  queryClient.prefetchQuery({ queryKey: ['document', nextDocId], queryFn: fetchDocument })

Progresso de carregamento: para documentos com 500+ seções, barra de progresso sutil (NProgress-style) no topo do viewer mostrando % de seções carregadas.

TOC interativa — modelo unificado de seção ativa:

  Estado: activeSectionIndex armazenado em useRef<number> (não useState — evitar re-renders da TOC a cada scroll frame). TOC lê via useSyncExternalStore ou callback subscrito ao ref.

  Fonte de verdade unificada:
  - Seções 0–19 (renderizadas no DOM): UM ÚNICO IntersectionObserver observando todos os headings, com rootMargin "-10% 0px -80% 0px". Callback atualiza activeSectionIndex.
  - Seções 20+ (virtualizadas): usar virtualizer.range.startIndex exposto por @tanstack/react-virtual. Debounce de 100ms no onScroll callback antes de atualizar activeSectionIndex.

  Reconciliação na zona de transição (seções 18–22):
  - Quando scrollTop está dentro de ±80px do offset da seção 20 (boundary de virtualização), APENAS o scroll-offset tracker é autoritativo. IntersectionObserver é temporariamente ignorado para índices nessa faixa.
  - Regra de prioridade: scroll-offset > IntersectionObserver quando ambos emitem dentro de 100ms.
  - Isso elimina flickering de TOC na transição entre estratégias.

  Navegação TOC → seção:
  - Seções 0–19 (no DOM): anchor links nativos com scroll-behavior: smooth.
  - Seções 20+ (virtualizadas):
    1. Chamar virtualizer.measure() para forçar cálculo de offsets.
    2. virtualizer.scrollToIndex(sectionIndex, { align: 'start' }).
