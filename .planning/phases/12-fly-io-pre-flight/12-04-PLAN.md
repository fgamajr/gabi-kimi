**Section: SEARCH API ARCHITECTURE**

Document the search API from the ULTRA MEGA PROMPT:

1. **Two search modes:**
   - **Citizen mode** (default, 90% of users): Queries use `summary_plain`, `key_facts`, and `category` fields. Results show plain-language summaries. Progressive disclosure: initially shows summary, expands to full text on click. Endpoint: `GET /api/v1/search?mode=citizen`
   - **Professional mode** (10% of users): Queries use full `body` text, `summary_technical`, and `references` fields. Results show technical summaries with legal citations. Endpoint: `GET /api/v1/search?mode=professional`

2. **Hybrid search with Reciprocal Rank Fusion (RRF):**
   - BM25 (Portuguese analyzer) provides keyword relevance
   - Dense vector (OpenAI embeddings) provides semantic relevance
   - RRF combines both rankings: `score = sum(1 / (k + rank_i))` with `k=60`
   - BM25 is ALWAYS available (baseline). Dense vectors are additive -- if embeddings are missing for some documents, those documents still appear via BM25
   - Fallback: If vector search fails (OpenAI outage), degrade to BM25-only transparently

3. **Conversational chat** (`/api/chat`):
   - Uses Qwen model (already implemented)
   - Rate-limited per user
   - Search-augmented: retrieves top-5 documents via hybrid search, passes as context to chat model

Note: The search API is ALREADY PARTIALLY IMPLEMENTED (BM25 search works). The citizen/professional mode split and RRF hybrid scoring are SPECIFICATIONS FOR FUTURE IMPLEMENTATION.

**Section: PROGRESSIVE DISCLOSURE UX PRINCIPLES**

Document the UX principles that affect pipeline and API design:

1. **Mobile-first**: All API responses must be optimized for mobile payload sizes. Summary fields (`summary_plain`) are returned by default; full `body` text only on explicit request (`?expand=body`).
2. **Citizen default**: The default search mode is citizen. Professional mode requires explicit opt-in. This means enrichment (Plan 03 enrichment modules) is a prerequisite for the full citizen experience.
3. **Two audiences**: Citizens (90%) want plain language, key facts, and relevance. Professionals (10%) want exact legal text, references, and affected entities. The pipeline must produce BOTH summary types during enrichment.
4. **Shareable content**: Every article must have a `shareable_text` field (280 chars) for social sharing. This is generated by the HighlightsGenerator enrichment module.

These principles are DESIGN CONSTRAINTS that affect how enrichment modules generate content and how the search API structures responses.

**Section: ADDITIONAL POSTGRES TABLES**

Document new Postgres tables from the ULTRA MEGA PROMPT (these extend the existing auth schema):

```sql
-- Auditor profiles: tracks professional users who need advanced features
CREATE TABLE auditor_profiles (
    id SERIAL PRIMARY KEY,
    user_id INTEGER NOT NULL REFERENCES auth.user(id),
    organization TEXT,
    cpf_cnpj TEXT,
    professional_role TEXT,
    search_mode TEXT DEFAULT 'citizen',
    created_at TIMESTAMPTZ DEFAULT NOW(),
    updated_at TIMESTAMPTZ DEFAULT NOW()
);

-- Workspaces: collections of documents for professional users
CREATE TABLE workspaces (
    id SERIAL PRIMARY KEY,
    user_id INTEGER NOT NULL REFERENCES auth.user(id),
    name TEXT NOT NULL,
    description TEXT,
    created_at TIMESTAMPTZ DEFAULT NOW(),
    updated_at TIMESTAMPTZ DEFAULT NOW()
);

-- Workspace documents: many-to-many between workspaces and DOU articles
CREATE TABLE workspace_documents (
    id SERIAL PRIMARY KEY,
    workspace_id INTEGER NOT NULL REFERENCES workspaces(id),
    document_id TEXT NOT NULL,  -- ES document ID
    added_at TIMESTAMPTZ DEFAULT NOW(),
    notes TEXT
);
```

Note: These tables are in Postgres (NOT SQLite). They extend the existing `auth.*` schema on the WEB app. The pipeline worker does NOT interact with these tables -- they are web-app-only for user features.

**Section: COST TRACKING AND BUDGET CONTROLS**

Document the cost framework from the ULTRA MEGA PROMPT:

1. **Per-module cost tracking:**
   - Enrichment (GPT-4o-mini): ~$0.50/day at 1000 articles/day
   - Embeddings (OpenAI): ~$0.13/1000 articles (ada-002 pricing)
   - Telegram alerts: free tier (< 30 messages/day)
   - Total estimated daily cost: < $1.00/day for normal operation

2. **Budget controls:**
   - Daily enrichment budget: configurable via `ENRICHMENT_DAILY_BUDGET_USD` env var (default: $2.00)
   - If daily spend exceeds budget, enrichment pauses until next day. Articles still get indexed (BM25) but without enrichment fields.
   - Monthly cost cap: `MONTHLY_COST_CAP_USD` env var (default: $30.00)
   - Cost tracking stored in SQLite: `CREATE TABLE cost_log (id INTEGER PRIMARY KEY, date TEXT, module TEXT, tokens_in INTEGER, tokens_out INTEGER, cost_usd REAL, created_at TEXT);`

3. **Cost-aware rules:**
   - NEVER run enrichment on historical backfill without explicit operator approval (could be thousands of articles)
   - Embeddings for historical corpus should be batched and rate-limited
   - All OpenAI API calls must log token usage for cost auditing

Mark this entire section as SPECIFICATION FOR FUTURE IMPLEMENTATION.
