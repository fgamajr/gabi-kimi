## FINAL VERIFICATION REPORT (Strict Coverage Phase)

### Root problems found
1. **Test harness (C)** – zero-kelvin was treating HTTP response codes incorrectly: `curl -w "%{http_code}"` output was sometimes captured as a 6-digit value (e.g. `200000`), so seed and discovery trigger were reported as FAIL despite HTTP 200.
2. **Evidência** – report JSON was written to `/tmp/gabi-zero-kelvin-report.json`, which could hit permission denied in some environments; evidence CSVs had no documented way to be generated from a single clean run.

### Fixes applied
1. **tests/zero-kelvin-test.sh**
   - Normalized all HTTP code captures to the first 3 characters (`"${var:0:3}"`) for seed, discovery trigger, fetch trigger, links, and ingest trigger, so spurious concatenation no longer causes false failures.
   - Default report path changed to `reports/zero-kelvin-report.json`; `mkdir -p` before writing the report.
2. **Tests for config propagation and fallback**
   - `tests/Gabi.Postgres.Tests/CatalogSeedStrictCoveragePropagationTests.cs`: seed with YAML `pipeline.coverage.strict: true` / `false` persists correctly into `PipelineConfig`.
   - `tests/Gabi.Api.Tests/DashboardStrictCoverageFallbackTests.cs`: triggering discovery/fetch without `strict_coverage` in the request enqueues jobs with `strict_coverage=true` when the source has `PipelineConfig.coverage.strict` in DB.
3. **FakeJobQueueRepository** – added `LastEnqueuedJob` and `ClearLastEnqueuedJob()` for payload assertions in API tests.
4. **CustomWebApplicationFactory** – added `EnsureSourceWithStrictPipelineConfigAsync` for fallback tests.
5. **scripts/export-strict-coverage-evidence.sh** – new script to export strict-coverage evidence CSVs from the current DB (strict sources from DB, last discovery/fetch run per source); to be run after zero-kelvin for clean evidence.

### Why the system is now correct
- **YAML → DB**: Seed persists full `pipeline` (including `coverage.strict`); unit tests assert it.
- **DB → Dashboard → Worker**: `ResolveStrictCoverage` reads `PipelineConfig` when the request omits `strict_coverage`; integration tests assert the enqueued job payload.
- **Discovery/Fetch semantics**: Existing code and `SourceDiscoveryJobExecutorMetadataTests` / `FetchJobExecutor` ensure strict + cap → inconclusive/capped and `Success=false`.
- **zero-kelvin**: Passes 15/15 with targeted run (e.g. `tcu_sumulas`, discovery, cap=50); strict_coverage is sent from YAML; inconclusive is FAIL; HTTP code handling is robust.
- **Evidência**: Report goes to `reports/zero-kelvin-report.json`; `export-strict-coverage-evidence.sh` produces CSVs from the current run when executed after zero-kelvin in the same environment.

### Remaining risks
- **Merge of defaults**: Seed does not apply `defaults.pipeline` from YAML; new sources must set `pipeline.coverage.strict` explicitly.
- **Worker externo**: Another Worker on the same DB can process jobs and alter runs; prefer a single API+Worker when validating.
- **Reprodutibilidade**: A second full zero-kelvin run with the same options was not re-executed in this session; the same command is expected to yield the same result (deterministic setup).
